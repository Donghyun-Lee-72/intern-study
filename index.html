무엇을 배우나요?

Section 0은 AI 개념을 배우기 전에 모두가 같은 출발선에 서도록 만드는 단계입니다. 수업에서 사용할 실행 기준은 Windows PowerShell, uv, Python 3.11, Jupyter Notebook입니다. Python은 브라우저에서 직접 설치하는 것을 기본 경로로 삼지 않습니다. uv가 필요한 Python 실행 환경과 패키지를 준비하고, 우리는 uv run --python 3.11 ... 형식으로 실행합니다.

Section 1: AI에게 요청 보내기

예상 60-90분 LLM API Prompt Token 감각

핵심 개념

LLM API는 AI 모델을 Python 코드에서 호출하는 인터페이스입니다. 채팅창에서 모델에게 말을 거는 것과 비슷하지만, 코드에서는 입력과 출력이 다음 단계로 이어져야 합니다. 입력은 prompt와 설정값이고, 출력은 텍스트 또는 구조화된 데이터입니다. 같은 prompt라도 모델 버전, 설정값, 입력 길이에 따라 답이 조금씩 달라질 수 있으므로, 결과를 그대로 믿는 대신 구조와 근거를 확인하는 습관이 필요합니다.

Prompt는 모델에게 주는 전체 지시입니다. 좋은 prompt에는 역할, 목표, 입력 자료, 제약 조건, 출력 형식, 실패 처리 규칙이 들어갑니다. 예를 들어 “요약해줘”보다 “아래 기술 노트에서 topic, objective, method, result, limitations, evidence를 JSON 형식으로 추출하고, 없는 정보는 not specified로 남겨라”가 더 좋은 지시입니다.

Token은 모델이 읽고 쓰는 글의 양을 세는 단위입니다. 입력 문서가 길수록 비용과 시간이 늘고, 모델이 한 번에 처리할 수 있는 범위도 제한됩니다. 그래서 긴 문서를 무작정 넣기보다, 필요한 부분을 찾고, 답변을 짧고 구조적으로 받는 설계가 중요합니다.

이번 Section에서 보는 예시

짧은 기술 노트 여러 개를 모델에게 보내고, 각 노트에서 핵심 정보를 추출합니다. 수업에서 중요한 것은 멋진 문장으로 요약하는 것이 아니라, 나중에 검증하고 재사용할 수 있는 구조로 정보를 꺼내는 것입니다. topic은 무엇에 대한 문서인지, objective는 목적이 무엇인지, method는 어떤 접근을 쓰는지, result는 어떤 결과가 있는지, evidence는 그 판단을 뒷받침하는 원문 조각이 무엇인지 확인합니다.

특히 문서에 없는 내용을 채워 넣지 않는 태도가 중요합니다. 모델이 답을 그럴듯하게 만들 수 있더라도, 입력에 없는 metric이나 validation plan은 not specified로 남겨야 합니다. 이 기준은 뒤의 RAG와 workflow에서도 그대로 이어집니다.

Section 2: 답변을 정해진 형식으로 받기

예상 60-90분 Structured output Pydantic LangChain parser

핵심 개념

자연어 답변은 사람이 읽기에는 좋지만, 프로그램의 다음 단계로 넘기기에는 불안정합니다. 필드 이름이 바뀌거나, 값이 빠지거나, 숫자가 문자열로 들어오면 다음 코드가 깨질 수 있습니다. Structured output은 모델 답변을 정해진 구조로 받는 방식이고, 이번 수업에서는 Pydantic으로 그 구조를 검사합니다.

Pydantic은 Python type hint를 사용해 데이터 구조와 validation 규칙을 정의하는 도구입니다. BaseModel은 어떤 필드가 필요한지 정하고, Field는 값의 범위나 설명을 붙입니다. 예를 들어 confidence가 0보다 작거나 1보다 크면 validation error가 나야 합니다. 이 오류는 실패라기보다 “모델 출력 또는 schema 계약을 다시 봐야 한다”는 디버깅 신호입니다.

중요한 구분이 있습니다. Pydantic validation은 형식을 검사하지만 사실성을 검사하지 않습니다. source_id와 quote 모양이 맞더라도, quote가 실제 원문에 없다면 사실성은 틀린 것입니다. Section 2에서는 형식을 안정화하고, Section 3에서 원문 근거를 확인합니다.

LangChain은 여기서 무엇을 하나요?

LangChain은 LLM application의 부품을 연결하는 프레임워크입니다. 이번 Section에서는 LangChain 전체를 깊게 배우지 않고, PromptTemplate, model call, output parser가 각각 어떤 역할을 하는지만 봅니다. PromptTemplate은 입력 문서와 형식 지시문을 합쳐 모델에게 보낼 prompt를 만들고, output parser는 모델이 만든 JSON 문자열을 Pydantic 객체로 바꿉니다.

LangChain을 쓴다고 답이 자동으로 좋아지지는 않습니다. 연결 도구는 prompt와 parser를 다루기 쉽게 해주지만, 여전히 schema 설계, evidence 확인, 답이 없을 때의 정책은 사람이 정해야 합니다.

Section 3: 문서 근거로 답하게 하기

예상 90-120분 RAG Evidence LangChain Document

핵심 개념

RAG는 Retrieval-Augmented Generation의 줄임말입니다. 모델이 기억만으로 바로 답하는 것이 아니라, 먼저 질문과 관련된 문서를 찾고, 그 문서 조각을 근거로 답변을 생성하는 패턴입니다. 검색 단계가 retrieval이고, 근거를 바탕으로 답을 쓰는 단계가 generation입니다.

RAG를 쓰는 이유는 모델이 모든 최신 문서와 내부 문서를 기억하고 있다고 가정하지 않기 위해서입니다. API onboarding 문서, support workflow 문서, release note 문서처럼 우리 프로젝트 안에 있는 자료를 먼저 찾고, 그 안에서만 답하게 만들면 근거 없는 답변을 줄일 수 있습니다. 하지만 RAG가 hallucination을 완전히 없애지는 못합니다. 검색된 문서가 틀리거나 질문과 관련이 없으면, 모델은 그 근거로도 틀린 답을 만들 수 있습니다.

그래서 RAG에서 가장 먼저 볼 것은 모델 답변이 아니라 retrieved evidence입니다. 어떤 문서가 검색되었는지, 그 문서가 질문에 직접 답하는지, source_id와 quote가 유지되는지 확인합니다. 답이 없는 질문이라면 억지로 답하지 않고 not_answered: true가 되어야 합니다.

Lexical retrieval과 실제 RAG의 차이

이번 실습은 초심자용이라 embedding/vector DB를 바로 쓰지 않습니다. 대신 단어가 얼마나 겹치는지 보는 lexical retrieval로 RAG의 구조를 먼저 익힙니다. 이 방식은 단순하고 이해하기 쉽지만, 표현이 많이 바뀐 질문에는 약합니다. 실제 프로젝트에서는 embedding model이 텍스트를 의미 벡터로 바꾸고, vector store가 유사한 chunk를 찾아주는 방식으로 확장할 수 있습니다.

LangChain 예제에서는 Document 객체와 metadata를 봅니다. 문서 본문만 들고 다니면 답변 출처를 잃기 쉽기 때문에, metadata.source_id에 문서 ID를 넣어 끝까지 유지합니다.

Section 4: 여러 단계를 하나로 연결하기

예상 60-90분 Workflow Tool Agent-like flow

핵심 개념

Workflow는 여러 단계를 순서대로 연결한 실행 흐름입니다. Section 4에서는 검색, 답변 생성, 사실 추출, schema validation, 로그 기록을 하나로 묶습니다. 여기서 만드는 것은 완전 자율 agent가 아니라 agent-like workflow입니다. AI를 사용하지만, 어떤 단계를 어떤 순서로 실행할지는 사람이 명시적으로 정합니다.

초심자에게는 이 방식이 더 안전합니다. 완전 자율 agent loop는 모델이 다음 행동을 직접 고르고, 실행 결과를 관찰한 뒤 다시 다음 행동을 고르는 구조입니다. 강력하지만 디버깅이 어렵습니다. 반면 고정 workflow는 각 단계의 입력과 출력이 보이므로, retrieval이 틀렸는지, 답변 생성이 틀렸는지, schema validation이 틀렸는지 나눠서 확인할 수 있습니다.

Tool은 workflow가 호출할 수 있는 외부 기능입니다. 이번 실습에서는 검색 함수, tokenization 함수, percentage extraction 같은 작은 Python 함수가 tool에 가깝습니다. 좋은 tool은 입력과 출력이 좁고 명확합니다. 너무 넓은 analyze_everything 같은 함수보다, 작고 검증 가능한 함수가 workflow를 안정적으로 만듭니다.

이번 Section의 최종 그림

Section 5: Linux/WSL2 개발 환경 입문

예상 90-120분 마지막 특강 WSL2 개발 환경 확장
Section 5는 Section 1-4의 AI 실습을 막지 않기 위해 마지막에 따로 둔 Linux 특강입니다. 기본 수업은 Windows PowerShell + uv로 진행하고, Linux/WSL2는 개발 환경을 넓히는 단계로 다룹니다.

왜 Linux를 배우나요?

많은 연구 코드, 서버 배포, 오픈소스 예제가 Linux 환경을 기준으로 설명됩니다. Windows만 써도 이번 AI 실습은 진행할 수 있지만, 이후 프로젝트에서 서버에 접속하거나, GitHub README의 bash 명령을 따라 하거나, Docker와 원격 개발 환경을 다루려면 Linux shell 감각이 필요합니다.

WSL2는 Windows 안에서 Ubuntu 같은 Linux 환경을 실행하게 해주는 기능입니다. 하지만 처음부터 WSL2를 필수로 넣으면 재부팅, Ubuntu 계정 생성, Windows/Linux 경로 차이, VS Code Remote - WSL 연결 같은 환경 문제가 AI 학습을 방해할 수 있습니다. 그래서 Section 5에서 별도로 다룹니다.

꼭 구분해야 할 것

PowerShell과 WSL Ubuntu shell은 서로 다른 실행 환경입니다. Windows 경로는 보통 C:\Users\...처럼 보이고, Linux 경로는 /home/...처럼 보입니다. WSL 안에서 Windows 파일은 /mnt/c/... 아래에 보일 수 있습니다. 같은 컴퓨터라도 어느 shell에서 실행했는지에 따라 Python 위치, 설치된 패키지, 현재 폴더가 달라질 수 있습니다.

Linux에서 가장 먼저 익힐 명령은 많지 않습니다. pwd는 현재 위치, ls는 파일 목록, cd는 폴더 이동, mkdir은 폴더 생성입니다. 오류가 났을 때는 “무슨 명령을 어디에서 실행했는가”가 중요하므로, 습관적으로 pwd와 ls를 확인합니다.

특강에서 실행할 것

05_linux_wsl.ipynb를 열어 Linux/WSL2 설명을 읽고, Python으로 현재 폴더를 확인하는 셀을 실행합니다. 아래 명령은 WSL Ubuntu shell에서 실행하는 예시입니다. Linux package 설치나 sudo, apt가 필요한 경우에는 먼저 이유를 설명하고 별도로 요청합니다. Python library는 Linux에서도 uv로 설치하고 실행합니다.

  • PowerShell과 WSL Ubuntu shell이 다른 실행 환경이라는 점을 설명할 수 있어야 합니다.
  • pwd, ls, cd가 각각 무엇을 확인하는지 말할 수 있어야 합니다.
  • Windows 경로와 Linux 경로가 다르게 보이는 이유를 이해해야 합니다.
  • VS Code Remote - WSL로 Linux 폴더를 여는 흐름을 설명할 수 있어야 합니다.
  • Linux에서도 uv/Python 실행 환경을 만들 수 있음을 확인하면 통과입니다.

API 키와 예상 비용

이 탭은 AI 모델을 쓰기 전에 알아야 할 “키와 사용량” 안내입니다. API는 내 코드가 OpenAI 모델에게 요청을 보내고 답을 받는 방식입니다. 요청할 때 사용하는 비밀번호 같은 값이 API key입니다. 이 key는 웹사이트나 채팅에 입력하지 않고, 각 노트북의 API key 입력 셀에만 붙여넣습니다.

왜 비용이 생기나요?

모델에게 보내는 입력과 모델이 생성하는 출력은 token이라는 사용량 단위로 계산됩니다. 처음에는 token을 “AI가 읽고 쓴 글의 양을 세는 단위” 정도로 이해하면 됩니다. 글을 많이 보내거나 긴 답변을 많이 받으면 사용량이 늘고, 사용량에 따라 비용이 생깁니다.

이번 수업은 제공받은 수업용 API key를 사용하는 흐름을 기준으로 합니다. 학생 개인 결제카드를 등록하지 않습니다. 별도로 안내받지 않았다면 개인 OpenAI 계정에 결제수단을 추가할 필요가 없습니다.
Section 0-4는 실제 생성 모델을 호출합니다. Section 5-6, 8, 10-12의 필수 실습은 API를 호출하지 않고, Section 7과 9는 기본 실습을 오프라인으로 실행한 뒤 선택적으로 작은 실제 API 검증을 수행합니다. 모델 응답 길이와 재실행 횟수에 따라 사용량이 늘 수 있으므로 실패한 셀만 다시 실행합니다.

비용 계산은 어떻게 보나요?

실제 금액은 사용하는 모델의 최신 가격표에 따라 달라집니다. 기본 계산식은 다음처럼 이해하면 됩니다. 

예상 비용 = input token 수 × input token 단가 + output token 수 × output token 단가
  • mini 계열 모델을 쓰면 학생 1명이 Section 노트북을 한 번씩 실행하는 비용은 작게 유지하는 것을 목표로 합니다.
  • 반복 실험, 긴 문서, 여러 번 재실행을 포함하면 비용이 늘어날 수 있습니다.
  • 정확한 금액은 실제 모델의 input/output token 단가와 재실행 횟수에 따라 달라집니다.

계산 예시

아래는 계산 방법을 보여주기 위한 가상의 단가 예시입니다. 가격표가 input 100만 token당 0.20달러, output 100만 token당 0.80달러라고 가정해 보겠습니다. 한 번 실행에서 input 3,000 token, output 1,500 token을 사용했다면 다음처럼 계산합니다. 

input 비용  = 3,000 / 1,000,000 × 0.20달러 = 0.0006달러
output 비용 = 1,500 / 1,000,000 × 0.80달러 = 0.0012달러
총 비용     = 0.0018달러

실제 단가는 수업에서 제공받은 모델의 최신 가격표를 기준으로 확인합니다. 예시 단가를 실제 청구액으로 보면 안 됩니다.
