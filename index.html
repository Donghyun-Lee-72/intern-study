Section 0: 시작 전 준비

예상 60-90분

핵심개념

Section 0의 목표는 AI 개념을 배우기 전에 “내 컴퓨터에서 같은 명령이 같은 방식으로 실행되는 상태”를 만드는 것입니다.

  • PowerShell은 Windows에서 명령을 입력하는 기본 터미널입니다.
  • uv는 Python 실행 환경과 패키지 설치를 관리하는 도구입니다.
  • Python은 브라우저에서 따로 받는 것이 아니라 uv run --python 3.12 ...로 실행합니다.
  • API key는 AI 모델을 호출할 수 있게 해주는 비밀번호 같은 값입니다.

강의자료

이 과정은 “웹사이트에서 읽기 → 노트북에서 실행 → 결과 기준과 비교” 순서로 진행합니다.

웹사이트는 교재이고, 노트북은 실행 도구입니다. API key는 웹사이트나 Python 코드에 직접 쓰지 않고 examples/.env 파일에만 둡니다.

smoke test는 모델이 좋은 답을 하는지 평가하는 단계가 아니라, key, model, network, package가 제대로 연결되는지 확인하는 단계입니다.

실습과제

  1. 실습 ZIP을 다운로드하고 압축을 풉니다.
  2. PowerShell에서 압축을 푼 폴더의 examples로 이동합니다.
  3. .env.example을 복사해 .env를 만들고 key/model을 넣습니다.
  4. 첫 연결 테스트를 실행합니다.
  5. 성공하면 노트북을 열고 Section 1로 넘어갑니다.

결과기준

  • uv --version이 동작합니다.
  • uv run --python 3.12 python --version이 동작합니다.
  • smoke test가 한국어 성공 문장을 출력합니다.
  • API key를 채팅, 제출물, 코드에 노출하지 않습니다.
  • 오류가 나면 실행한 명령과 오류 메시지만 공유할 수 있습니다.
cd amsl-internship-study/examples
Copy-Item .env.example .env
notepad .env
uv run --python 3.12 --with-requirements requirements.txt python live_openai_smoke.py
실습 파일 ZIP 다운로드 노트북 다운로드

Section 1: AI에게 요청 보내기

예상 60-90분

핵심개념

LLM API는 AI 모델을 내 코드에서 함수처럼 호출하는 방법입니다. 입력은 prompt이고, 출력은 모델이 생성한 답변입니다.

  • Prompt는 모델에게 주는 작업 지시문입니다.
  • 좋은 prompt에는 역할, 목표, 입력 자료, 제약 조건, 출력 형식이 들어갑니다.
  • Token은 모델이 읽고 쓰는 글의 양을 세는 단위입니다.
  • 같은 prompt라도 모델 답변은 조금씩 달라질 수 있습니다.

강의자료

Section 1에서는 “AI가 그럴듯한 문장을 만드는 것”보다 “내가 원하는 정보 구조로 답하게 만드는 것”에 집중합니다.

기술 노트에서 topic, objective, method, result, evidence를 뽑는 이유는 나중에 검색, 검증, 보고서 작성에 재사용하기 위해서입니다.

문서에 없는 내용은 채워 넣지 않고 not specified처럼 모른다고 남기는 것이 더 좋은 답입니다.

실습과제

  1. 노트북에서 API key/model 설정 셀을 확인합니다.
  2. Section 1 셀을 실행해 첫 live API 응답을 받습니다.
  3. 기술 노트에서 핵심 필드가 어떻게 추출되는지 봅니다.
  4. prompt를 읽고 역할, 목표, 입력, 제약, 출력 형식을 표시해 봅니다.
uv run --python 3.12 --with-requirements requirements.txt python section1_llm_api.py

결과기준

  • 입력 문서 ID가 evidence에 유지됩니다.
  • 문서에 없는 내용은 not specified로 남습니다.
  • 좋은 prompt 요소 5개를 설명할 수 있습니다.
  • confidence가 0과 1 사이 숫자로 나옵니다.
  • 답변 문장이 조금 달라도 구조가 유지되면 통과로 봅니다.

Section 2: 답변을 정해진 형식으로 받기

예상 60-90분

핵심개념

Structured output은 모델 답변을 자유로운 문장이 아니라 정해진 데이터 구조로 받는 방식입니다.

  • Pydantic은 Python type hint를 이용해 데이터 구조를 정의하고 검사합니다.
  • Validation은 필수 필드, 타입, 값 범위를 확인합니다.
  • Parser는 모델이 만든 텍스트를 Python 객체로 바꾸는 역할을 합니다.
  • LangChain은 prompt, model, parser 같은 부품을 연결하는 프레임워크입니다.

강의자료

중요한 구분은 “형식 검증”과 “사실 검증”입니다. Pydantic validation을 통과했다는 말은 데이터 모양이 맞다는 뜻이지, 내용이 참이라는 뜻은 아닙니다.

예를 들어 confidence가 문자열이면 형식 오류이고, 원문에 없는 quote를 넣으면 사실성 오류입니다. Section 2는 먼저 형식을 안정화하고, Section 3에서 근거를 확인합니다.

LangChain의 PydanticOutputParser는 모델에게 원하는 JSON 형식을 알려주고, 받은 결과를 schema에 맞춰 해석합니다.

실습과제

  1. schema.py가 정의하는 필드와 타입을 봅니다.
  2. 일부러 잘못된 데이터를 넣었을 때 validation이 실패하는지 확인합니다.
  3. Section 1의 모델 출력이 Pydantic 객체로 바뀌는 흐름을 확인합니다.
  4. LangChain prompt/parser 예제에서 prompt preview, live text, parsed object의 차이를 비교합니다.
uv run --python 3.12 --with-requirements requirements.txt python schema.py
uv run --python 3.12 --with-requirements requirements.txt python section2_structured_output.py
uv run --python 3.12 --with-requirements requirements.txt python section2_langchain_contract.py

결과기준

  • 필수 필드가 빠지면 validation이 실패합니다.
  • confidence는 0과 1 사이입니다.
  • 형식 검증과 사실 확인의 차이를 설명할 수 있습니다.
  • evidence는 문자열 목록이 아니라 source_id와 quote를 가진 객체 목록입니다.
  • LangChain을 쓴다고 자동으로 사실성이 보장되지는 않는다는 점을 이해합니다.

Section 3: 문서 근거로 답하게 하기

예상 90-120분

핵심개념

RAG는 Retrieval-Augmented Generation의 줄임말입니다. 질문에 바로 답하지 않고, 먼저 관련 문서를 찾은 뒤 그 문서를 근거로 답합니다.

  • Retrieval은 질문과 관련 있는 문서 조각을 찾는 단계입니다.
  • Generation은 찾은 근거를 바탕으로 답변을 쓰는 단계입니다.
  • Evidence는 답변을 뒷받침하는 원문 출처와 인용문입니다.
  • Chunking은 긴 문서를 작은 조각으로 나누는 과정입니다.

강의자료

RAG의 첫 확인 대상은 모델 답변이 아니라 검색된 근거입니다. 검색된 문서가 틀리면 모델은 그럴듯하지만 틀린 답을 만들 수 있습니다.

이번 실습은 초심자용이라 embedding/vector DB 대신 단어 기반 lexical retrieval로 시작합니다. 구조를 이해한 뒤 실제 프로젝트에서는 embedding similarity와 vector store retriever로 바꿀 수 있습니다.

RAG를 쓰면 hallucination이 줄어들 수 있지만 완전히 사라지지는 않습니다. 그래서 source_id, quote, not_answered 정책을 함께 봅니다.

실습과제

  1. 질문별로 어떤 문서가 검색되는지 먼저 확인합니다.
  2. 검색된 문서 안에서 답변에 쓰인 quote를 직접 찾아봅니다.
  3. 답이 없는 질문에서 not_answered가 켜지는지 확인합니다.
  4. LangChain 문서 객체와 metadata가 source id를 어떻게 들고 가는지 확인합니다.
uv run --python 3.12 --with-requirements requirements.txt python section3_rag.py
uv run --python 3.12 --with-requirements requirements.txt python section3_langchain_rag.py

결과기준

  • 답변은 검색된 문서의 quote를 근거로 삼습니다.
  • 근거가 없으면 not_answered: true가 됩니다.
  • keyword 검색과 embedding 검색의 차이를 말할 수 있습니다.
  • API onboarding 질문은 onboarding 문서를 근거로 답합니다.
  • 검색 결과에 잡힌 문서와 최종 답변 근거로 쓸 문서를 구분할 수 있습니다.

Section 4: 여러 단계를 하나로 연결하기

예상 60-90분

핵심개념

Workflow는 여러 단계를 순서대로 연결한 실행 흐름입니다. Section 4에서는 검색, 답변 생성, 사실 추출, 검증, 로그 기록을 하나로 묶습니다.

  • Tool은 workflow가 호출할 수 있는 외부 기능입니다.
  • State는 workflow가 현재 들고 있는 입력, 검색 결과, 중간 출력, 로그입니다.
  • Agent-like workflow는 AI를 쓰지만 단계 순서는 사람이 고정한 구조입니다.
  • 완전 자율 agent loop는 모델이 다음 행동을 직접 고르는 구조입니다.

강의자료

초심자 실습에서는 완전 자율 agent보다 고정 workflow가 안전합니다. 각 단계의 입력과 출력이 보여야 어디서 틀렸는지 찾을 수 있기 때문입니다.

이번 workflow는 질문을 받으면 관련 문서를 찾고, 근거 기반 답변을 만들고, 답변에서 핵심 사실을 뽑고, schema로 검증한 뒤 실행 로그를 남깁니다.

나중에 agent로 확장할 때도 먼저 “어떤 단계가 tool로 분리될 수 있는지”, “무엇은 사람이 고정해야 하는지”를 판단해야 합니다.

실습과제

  1. Section 4 셀을 실행해 전체 workflow 결과를 봅니다.
  2. 성공 질문과 답이 없는 질문의 차이를 비교합니다.
  3. log에서 retrieval, answer, fact extraction, validation 순서를 확인합니다.
  4. 작은 변형 질문을 하나 넣고 어디서 결과가 달라지는지 봅니다.
uv run --python 3.12 --with-requirements requirements.txt python section4_workflow.py

결과기준

  • answer_ko, facts, evidence, log가 출력됩니다.
  • facts는 같은 evidence에서 나옵니다.
  • 답이 없는 질문은 낮은 confidence로 처리됩니다.
  • 각 단계가 실패했을 때 전체 workflow가 어디까지 진행됐는지 로그로 볼 수 있습니다.
  • agent와 workflow의 차이를 “누가 다음 행동을 고르는가”로 설명할 수 있습니다.

Section 5: Linux/WSL2 개발 환경 입문

예상 90-120분
Section 5는 노트북 필수 셀이 아니라, Section 1-4 이후에 따로 다루는 개발 환경 섹션입니다.

핵심개념

Section 5는 AI 필수 실습이 아니라 개발 환경 확장 섹션입니다. Windows에서 기본 실습을 끝낸 뒤 Linux/WSL2를 따로 배웁니다.

  • Linux는 서버와 연구 코드에서 많이 쓰이는 운영체제 계열입니다.
  • Ubuntu는 Linux 배포판 중 하나입니다.
  • WSL2는 Windows 안에서 Ubuntu 같은 Linux 환경을 실행하게 해줍니다.
  • Terminal, shell, bash는 같은 말이 아니므로 구분해서 봅니다.

강의자료

Linux에서는 경로 표기, 패키지 설치, Python 위치가 Windows와 다를 수 있습니다. 그래서 “지금 PowerShell인지, WSL Ubuntu shell인지”를 먼저 확인하는 습관이 중요합니다.

Windows 경로는 보통 C:\Users\...처럼 보이고, Linux 경로는 /home/...처럼 보입니다. WSL에서는 Windows 파일도 /mnt/c/... 아래에서 볼 수 있습니다.

VS Code Remote - WSL은 Windows의 VS Code로 Linux 폴더를 열어 개발할 수 있게 해주는 확장 흐름입니다.

실습과제

  1. PowerShell과 WSL Ubuntu shell을 구분합니다.
  2. pwd, ls, cd, mkdir로 파일 위치를 확인합니다.
  3. VS Code에서 WSL 폴더를 여는 흐름을 봅니다.
  4. Linux에서도 uv와 Python 실행을 확인합니다.
pwd
ls
cd ~
mkdir amsl-test
curl -LsSf https://astral.sh/uv/install.sh | sh
uv --version
uv run --python 3.12 python --version

결과기준

  • PowerShell과 WSL Ubuntu shell의 차이를 설명합니다.
  • Windows 경로와 Linux 경로가 다름을 압니다.
  • Linux에서도 uv/Python 환경을 만들 수 있음을 확인합니다.
  • 어느 위치에서 명령을 실행했는지 pwd로 확인할 수 있습니다.
  • Linux 환경 문제가 AI 실습 문제와 섞이지 않도록 분리해서 질문할 수 있습니다.

API 키와 예상 비용

이 탭은 AI 모델을 쓰기 전에 알아야 할 “키와 사용량” 안내입니다. API는 내 코드가 OpenAI 모델에게 요청을 보내고 답을 받는 방식입니다. 요청할 때 사용하는 비밀번호 같은 값이 API key입니다. 이 key는 웹사이트에 입력하지 않고, 내 컴퓨터의 .env 파일에만 저장합니다.

왜 비용이 생기나요?

모델에게 보내는 입력과 모델이 생성하는 출력은 token이라는 사용량 단위로 계산됩니다. 처음에는 token을 “AI가 읽고 쓴 글의 양을 세는 단위” 정도로 이해하면 됩니다. 글을 많이 보내거나 긴 답변을 많이 받으면 사용량이 늘고, 사용량에 따라 비용이 생깁니다.

이번 수업은 제공받은 수업용 API key를 사용하는 흐름을 기준으로 합니다. 학생 개인 결제카드를 등록하지 않습니다. 별도로 안내받지 않았다면 개인 OpenAI 계정에 결제수단을 추가할 필요가 없습니다.
수업용 전체 노트북 1회 실행은 정상 경로 기준 약 12회 요청, 약 4,500 token 정도로 측정되었습니다. JSON 형식 수정 재시도, 모델 응답 길이, 재실행 횟수에 따라 더 늘 수 있으므로 여유 있게 1만 token 안팎으로 생각하면 됩니다.

비용 계산은 어떻게 보나요?

실제 금액은 사용하는 모델의 최신 가격표에 따라 달라집니다. 기본 계산식은 다음처럼 이해하면 됩니다. 

예상 비용 = input token 수 × input token 단가 + output token 수 × output token 단가
  • mini 계열 모델을 쓰면 학생 1명이 전체 노트북을 한 번 끝까지 실행하는 비용은 작게 유지하는 것을 목표로 합니다.
  • 반복 실험, 긴 문서, 여러 번 재실행을 포함하면 비용이 늘어날 수 있습니다.
  • 정확한 금액은 실제 모델의 input/output token 단가와 재실행 횟수에 따라 달라집니다.

계산 예시

아래는 계산 방법을 보여주기 위한 가상의 단가 예시입니다. 가격표가 input 100만 token당 0.20달러, output 100만 token당 0.80달러라고 가정해 보겠습니다. 한 번 실행에서 input 3,000 token, output 1,500 token을 사용했다면 다음처럼 계산합니다. 

input 비용  = 3,000 / 1,000,000 × 0.20달러 = 0.0006달러
output 비용 = 1,500 / 1,000,000 × 0.80달러 = 0.0012달러
총 비용     = 0.0018달러

실제 단가는 수업에서 제공받은 모델의 최신 가격표를 기준으로 확인합니다. 예시 단가를 실제 청구액으로 보면 안 됩니다. 노트북 마지막 셀에는 input/output token 수를 기록하는 칸이 있습니다.

비용을 줄이는 규칙

  • 긴 PDF 전체를 넣지 않고, 짧은 텍스트 조각으로 실습합니다.
  • 실습에 넣는 문서 수와 질문 수를 제한합니다.
  • 실패한 셀만 다시 실행하고 전체 노트북 재실행은 최소화합니다.
  • 출력 토큰이 불필요하게 길어지지 않도록 답변 형식을 짧고 구조적으로 요청합니다.

실습 후 기록할 것

  • 사용한 모델명
  • 실행한 노트북 섹션
  • 대략적인 input/output token 수
  • 실패해서 다시 실행한 셀

결과 확인: 실행 후 무엇을 봐야 하나요?

AI 실습에서는 문장이 매번 완전히 같을 필요가 없습니다. 대신 결과가 정해진 구조를 지키는지, 근거가 실제 입력 문서에서 왔는지, 답이 없는 질문을 억지로 지어내지 않는지를 확인합니다. 노트북을 실행한 뒤 이 탭과 비교하면 됩니다.

Section 1: API와 prompt

  • topic, objective, method, result가 비어 있지 않은지 확인합니다.
  • evidence의 source_id가 입력 문서 ID와 같은지 확인합니다.
  • confidence가 0과 1 사이 숫자인지 확인합니다.

Section 2: 형식 확인

  • 모델 문장이 조금 달라도 정해진 데이터 구조로 바뀌는지 확인합니다.
  • 필수 필드가 빠지거나 빈 문자열이면 실패해야 합니다.
  • 검증 통과 여부가 “답이 자연스러운가”보다 먼저입니다.

Section 3: 문서 근거 확인

  • 답변이 검색된 문서의 문장 또는 chunk를 근거로 삼는지 확인합니다.
  • 검색된 문서에 답이 없으면 not_answered가 true가 되어야 합니다.
  • 근거 없는 문장이 답변에 섞이면 실패로 봅니다.

Section 4: 단계 연결 확인

  • 최종 출력에 answer_ko, facts, evidence, log가 있어야 합니다.
  • facts 안의 근거는 앞 단계에서 확인된 evidence 안에서만 나와야 합니다.
  • 실행 로그가 입력, 검색, 추출, 검증 순서를 보여야 합니다.

Section 5: Linux/WSL2 환경 확인

  • PowerShell과 WSL Ubuntu shell이 다른 환경이라는 점을 설명할 수 있어야 합니다.
  • pwd, ls, cd로 현재 위치와 파일 목록을 확인할 수 있어야 합니다.
  • Windows 파일 경로와 Linux 파일 경로가 다를 수 있음을 알아야 합니다.
같은 셀을 다시 실행했을 때 표현이 조금 달라질 수 있습니다. 비교 기준은 “정확히 같은 문장”이 아니라 “같은 구조, 같은 근거, 같은 실패 처리 정책”입니다.

문제 해결

키 오류

OPENAI_API_KEY가 비어 있거나 잘못된 경우입니다. .env에 키를 한 줄로 넣고 터미널을 다시 엽니다.

모델 오류

gpt-5.4-mini는 현재 검증된 수업용 기본 모델명입니다. 계정에서 사용할 수 없으면 제공된 저비용 모델명으로 OPENAI_MODEL을 바꿉니다.

패키지 오류

먼저 uv run --python 3.12 --with-requirements requirements.txt python live_openai_smoke.py로 실행합니다. 이 방식은 가상환경 활성화가 헷갈려도 필요한 패키지를 함께 사용합니다.
