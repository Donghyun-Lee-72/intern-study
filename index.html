Agentic AI 실습형 학습 포털

이 수업은 AI 모델을 직접 불러와 보고, 답변을 확인하고, 근거가 있는 작은 AI 프로그램을 만들어보는 섹션형 실습입니다. 처음부터 모든 용어를 알 필요는 없습니다. 웹사이트에서 설명을 읽고, 노트북에서 코드를 실행하고, 결과 확인 페이지에서 맞게 실행됐는지 비교하면 됩니다.

처음이면 Section 0부터 Python 기본 문법이면 시작 가능 Windows PowerShell + uv 웹사이트는 교재 노트북은 실행용

학습 섹션

Section을 하나씩 펼쳐서 읽고, 노트북에서 같은 Section을 실행한 뒤, 바로 아래 확인 기준과 비교합니다. 별도 문서를 여러 개 오갈 필요 없이 이 화면만 따라가면 됩니다.

아직 RAG, Pydantic, LangChain, token 같은 단어를 몰라도 괜찮습니다. 지금은 “준비하기 → AI 호출하기 → 형식 확인하기 → 근거 확인하기 → 단계 연결하기”만 기억하면 됩니다. 어려운 용어는 필요한 섹션에서 하나씩 설명합니다.

Section 0: 시작 전 준비

예상 60-90분

핵심개념

이번 수업은 Windows PowerShell과 uv를 기준으로 진행합니다. Python은 직접 따로 설치하기보다 uv run --python 3.12 ...로 실행합니다.

강의자료

웹사이트는 교재, 노트북은 실행용입니다. API key는 웹사이트나 코드에 넣지 않고 examples/.env에만 둡니다.

  1. ZIP을 받고 압축을 풉니다.
  2. .env를 만들고 key/model을 넣습니다.
  3. 첫 연결 테스트를 실행합니다.

결과기준

  • uv --version이 동작합니다.
  • uv run --python 3.12 python --version이 동작합니다.
  • smoke test가 한국어 성공 문장을 출력합니다.
cd amsl-internship-study/examples

Section 1: AI에게 요청 보내기

예상 60-90분

핵심개념

LLM API는 모델을 함수처럼 호출하는 방법입니다. Prompt에는 역할, 목표, 입력, 제약, 출력 형식이 들어갑니다.

강의자료

모델 출력은 매번 조금 달라질 수 있습니다. 그래서 답변 문장보다 구조, 근거, 모르는 정보 처리 방식을 봅니다.

실습과제

노트북의 Section 1 셀을 실행해 기술 노트에서 topic, objective, method, result, evidence를 뽑습니다.

결과기준

  • 입력 문서 ID가 evidence에 유지됩니다.
  • 문서에 없는 내용은 not specified로 남습니다.
  • 좋은 prompt 요소 5개를 설명할 수 있습니다.

Section 2: 답변을 정해진 형식으로 받기

예상 60-90분

핵심개념

Structured output은 모델 답변을 정해진 구조로 받는 방식입니다. Pydantic은 타입, 필수 필드, 값 범위를 검사합니다.

강의자료

검증 통과는 “형식이 맞다”는 뜻이지 “사실이다”라는 뜻은 아닙니다. 사실 여부는 evidence와 원문으로 다시 봅니다.

실습과제

노트북의 Section 2 셀을 실행해 Pydantic schema와 LangChain parser가 모델 출력을 어떻게 확인하는지 봅니다.

결과기준

  • 필수 필드가 빠지면 validation이 실패합니다.
  • confidence는 0과 1 사이입니다.
  • 형식 검증과 사실 확인의 차이를 설명할 수 있습니다.

Section 3: 문서 근거로 답하게 하기

예상 90-120분

핵심개념

RAG는 질문과 관련된 문서를 먼저 찾고, 그 근거 안에서 답하게 하는 흐름입니다.

강의자료

검색된 문서가 틀리면 답도 틀릴 수 있습니다. 먼저 retrieved evidence를 보고, 그 다음 모델 답변을 봅니다.

실습과제

노트북의 Section 3 셀을 실행해 source id와 quote가 답변에 유지되는지 확인합니다.

결과기준

  • 답변은 검색된 문서의 quote를 근거로 삼습니다.
  • 근거가 없으면 not_answered: true가 됩니다.
  • keyword 검색과 embedding 검색의 차이를 말할 수 있습니다.

Section 4: 여러 단계를 하나로 연결하기

예상 60-90분

핵심개념

Workflow는 검색, 답변 생성, 사실 추출, 검증을 순서대로 연결한 작은 프로그램입니다.

강의자료

초심자 실습에서는 완전 자율 agent보다 고정 workflow가 안전합니다. 각 단계의 입력과 출력이 보여야 디버깅할 수 있습니다.

실습과제

노트북의 Section 4 셀을 실행하고 성공 질문과 실패 질문의 결과를 비교합니다.

결과기준

  • answer_ko, facts, evidence, log가 출력됩니다.
  • facts는 같은 evidence에서 나옵니다.
  • 답이 없는 질문은 낮은 confidence로 처리됩니다.

Section 5: Linux/WSL2 개발 환경 입문

예상 90-120분
Section 5는 노트북 필수 셀이 아니라, Section 1-4 이후에 따로 다루는 개발 환경 섹션입니다.

핵심개념

Windows PowerShell과 Linux shell은 서로 다른 실행 환경입니다. WSL2는 Windows 안에서 Linux 개발 공간을 쓰게 해줍니다.

강의자료

Linux에서는 경로, 패키지, Python 위치가 Windows와 다를 수 있습니다. 지금 어느 shell에서 실행 중인지 확인하는 습관이 중요합니다.

실습과제

pwd, ls, cd, mkdir, python3 --version을 실행하고 VS Code Remote - WSL 흐름을 확인합니다.

결과기준

  • PowerShell과 WSL Ubuntu shell의 차이를 설명합니다.
  • Windows 경로와 Linux 경로가 다름을 압니다.
  • Linux에서도 uv/Python 환경을 만들 수 있음을 확인합니다.

API 키와 예상 비용

이 탭은 AI 모델을 쓰기 전에 알아야 할 “키와 사용량” 안내입니다. API는 내 코드가 OpenAI 모델에게 요청을 보내고 답을 받는 방식입니다. 요청할 때 사용하는 비밀번호 같은 값이 API key입니다. 이 key는 웹사이트에 입력하지 않고, 내 컴퓨터의 .env 파일에만 저장합니다.

왜 비용이 생기나요?

모델에게 보내는 입력과 모델이 생성하는 출력은 token이라는 사용량 단위로 계산됩니다. 처음에는 token을 “AI가 읽고 쓴 글의 양을 세는 단위” 정도로 이해하면 됩니다. 글을 많이 보내거나 긴 답변을 많이 받으면 사용량이 늘고, 사용량에 따라 비용이 생깁니다.

이번 수업은 제공받은 수업용 API key를 사용하는 흐름을 기준으로 합니다. 학생 개인 결제카드를 등록하지 않습니다. 별도로 안내받지 않았다면 개인 OpenAI 계정에 결제수단을 추가할 필요가 없습니다.
수업용 전체 노트북 1회 실행은 정상 경로 기준 약 12회 요청, 약 4,500 token 정도로 측정되었습니다. JSON 형식 수정 재시도, 모델 응답 길이, 재실행 횟수에 따라 더 늘 수 있으므로 여유 있게 1만 token 안팎으로 생각하면 됩니다.

비용 계산은 어떻게 보나요?

실제 금액은 사용하는 모델의 최신 가격표에 따라 달라집니다. 기본 계산식은 다음처럼 이해하면 됩니다. 

예상 비용 = input token 수 × input token 단가 + output token 수 × output token 단가
  • mini 계열 모델을 쓰면 학생 1명이 전체 노트북을 한 번 끝까지 실행하는 비용은 작게 유지하는 것을 목표로 합니다.
  • 반복 실험, 긴 문서, 여러 번 재실행을 포함하면 비용이 늘어날 수 있습니다.
  • 정확한 금액은 실제 모델의 input/output token 단가와 재실행 횟수에 따라 달라집니다.

계산 예시

아래는 계산 방법을 보여주기 위한 가상의 단가 예시입니다. 가격표가 input 100만 token당 0.20달러, output 100만 token당 0.80달러라고 가정해 보겠습니다. 한 번 실행에서 input 3,000 token, output 1,500 token을 사용했다면 다음처럼 계산합니다. 

input 비용  = 3,000 / 1,000,000 × 0.20달러 = 0.0006달러
output 비용 = 1,500 / 1,000,000 × 0.80달러 = 0.0012달러
총 비용     = 0.0018달러

실제 단가는 수업에서 제공받은 모델의 최신 가격표를 기준으로 확인합니다. 예시 단가를 실제 청구액으로 보면 안 됩니다. 노트북 마지막 셀에는 input/output token 수를 기록하는 칸이 있습니다.

비용을 줄이는 규칙

  • 긴 PDF 전체를 넣지 않고, 짧은 텍스트 조각으로 실습합니다.
  • 실습에 넣는 문서 수와 질문 수를 제한합니다.
  • 실패한 셀만 다시 실행하고 전체 노트북 재실행은 최소화합니다.
  • 출력 토큰이 불필요하게 길어지지 않도록 답변 형식을 짧고 구조적으로 요청합니다.

실습 후 기록할 것

  • 사용한 모델명
  • 실행한 노트북 섹션
  • 대략적인 input/output token 수
  • 실패해서 다시 실행한 셀

결과 확인: 실행 후 무엇을 봐야 하나요?

AI 실습에서는 문장이 매번 완전히 같을 필요가 없습니다. 대신 결과가 정해진 구조를 지키는지, 근거가 실제 입력 문서에서 왔는지, 답이 없는 질문을 억지로 지어내지 않는지를 확인합니다. 노트북을 실행한 뒤 이 탭과 비교하면 됩니다.

Section 1: API와 prompt

  • topic, objective, method, result가 비어 있지 않은지 확인합니다.
  • evidence의 source_id가 입력 문서 ID와 같은지 확인합니다.
  • confidence가 0과 1 사이 숫자인지 확인합니다.

Section 2: 형식 확인

  • 모델 문장이 조금 달라도 정해진 데이터 구조로 바뀌는지 확인합니다.
  • 필수 필드가 빠지거나 빈 문자열이면 실패해야 합니다.
  • 검증 통과 여부가 “답이 자연스러운가”보다 먼저입니다.

Section 3: 문서 근거 확인

  • 답변이 검색된 문서의 문장 또는 chunk를 근거로 삼는지 확인합니다.
  • 검색된 문서에 답이 없으면 not_answered가 true가 되어야 합니다.
  • 근거 없는 문장이 답변에 섞이면 실패로 봅니다.

Section 4: 단계 연결 확인

  • 최종 출력에 answer_ko, facts, evidence, log가 있어야 합니다.
  • facts 안의 근거는 앞 단계에서 확인된 evidence 안에서만 나와야 합니다.
  • 실행 로그가 입력, 검색, 추출, 검증 순서를 보여야 합니다.

Section 5: Linux/WSL2 환경 확인

  • PowerShell과 WSL Ubuntu shell이 다른 환경이라는 점을 설명할 수 있어야 합니다.
  • pwd, ls, cd로 현재 위치와 파일 목록을 확인할 수 있어야 합니다.
  • Windows 파일 경로와 Linux 파일 경로가 다를 수 있음을 알아야 합니다.
같은 셀을 다시 실행했을 때 표현이 조금 달라질 수 있습니다. 비교 기준은 “정확히 같은 문장”이 아니라 “같은 구조, 같은 근거, 같은 실패 처리 정책”입니다.

문제 해결

키 오류

OPENAI_API_KEY가 비어 있거나 잘못된 경우입니다. .env에 키를 한 줄로 넣고 터미널을 다시 엽니다.

모델 오류

gpt-5.4-mini는 현재 검증된 수업용 기본 모델명입니다. 계정에서 사용할 수 없으면 제공된 저비용 모델명으로 OPENAI_MODEL을 바꿉니다.

패키지 오류

먼저 uv run --python 3.12 --with-requirements requirements.txt python live_openai_smoke.py로 실행합니다. 이 방식은 가상환경 활성화가 헷갈려도 필요한 패키지를 함께 사용합니다.

결과가 흔들림

AI 출력은 약간 변할 수 있습니다. 정해진 형식 확인 실습에서는 validation을 통과하는지가 더 중요합니다.
