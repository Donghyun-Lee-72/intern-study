Section 7. Embedding과 VectorStore로 의미 검색하기

예상 시간: 필수 120~150분 / 선택 심화 30~60분
선수 과정: Section 6
API 비용: 필수 offline 실습은 없음 / 실제 embedding 선택 실습은 호출 전 문서·질문 수 확인

이번 Section의 위치

Section 3의 lexical retrieval은 질문과 문서에서 겹치는 단어를 찾았습니다. Section 6에서는 긴 문서를 chunk로 나누고 source_id, chunk_id, 원문 위치를 보존했습니다. 이제 각 chunk를 숫자 벡터로 바꾸고, 질문과 의미상 가까운 chunk를 찾습니다.

이번 목표는 특정 Vector DB 제품을 익히는 것이 아닙니다. 다음 네 단계를 분리해 이해하는 것이 핵심입니다.

text → embedding vector → similarity 계산 → 관련 chunk 반환

핵심 질문

질문과 문서에 같은 단어가 거의 없어도 관련 내용을 찾을 수 있는 이유는 무엇이며, 그 결과를 어디까지 믿어야 할까요?

학습 목표

  1. embedding을 텍스트의 비교 가능한 특성을 담은 숫자 배열로 설명합니다.
  2. query와 document가 같은 embedding 공간에서 비교되어야 하는 이유를 설명합니다.
  3. cosine similarity를 방향의 유사성이라는 수준에서 계산하고 해석합니다.
  4. Section 6의 chunk를 in-memory VectorStore에 저장하고 top-k 검색을 수행합니다.
  5. lexical 검색과 vector 검색의 성공·실패 사례를 비교합니다.
  6. similarity가 높다는 사실과 답이 맞다는 사실을 구분합니다.

이번 Section의 필수선과 발전선

  • 필수선: 2차원 예제로 cosine similarity를 계산하고, 같은 질문에 lexical/vector 검색 결과가 달라지는 이유를 설명합니다.
  • 발전선: 실제 embedding API, 영구 VectorStore, 여러 모델 비교는 선택입니다.
  • 복습 신호: 벡터의 내적·크기 또는 Python의 list[float]가 낯설다면 공식보다 먼저 본문의 숫자 예제를 따라갑니다.

AI 전공 과목에서 embedding을 이미 배웠더라도 이 장을 건너뛰지는 않습니다. 목적은 수학을 다시 배우는 것이 아니라 모델·분할 설정·corpus 버전이 index와 함께 관리되어야 한다는 구현 계약을 익히는 데 있습니다. 선형대수 증명은 이 과정의 통과 조건이 아닙니다.

1. 선수지식 확인

  • Python의 list[float]가 어떤 값인지 설명할 수 있나요?
  • 두 점 사이의 거리가 가까우면 항상 같은 의미라고 말할 수 있을까요?
  • Section 6의 source_id와 chunk_id가 검색 결과에도 남아야 하는 이유는 무엇인가요?
  • 질문에 답하는 문서가 corpus에 없다면 vector 검색은 무엇을 반환할까요?

2. Embedding은 무엇인가요?

Embedding model은 텍스트를 고정된 길이의 숫자 배열로 변환합니다. 이 벡터의 개별 숫자를 사람이 “첫 번째 값은 보안, 두 번째 값은 비용”처럼 직접 해석하지는 않습니다. 대신 같은 model로 만든 벡터끼리 거리를 비교하여 텍스트 사이의 관련성을 추정합니다.

"API key를 공개하면 안 됩니다"  → [ 0.12, -0.41, ... ]
"비밀 키를 저장소에 올려도 되나요?" → [ 0.09, -0.37, ... ]

두 문장은 같은 단어가 많지 않지만, 적절한 embedding model에서는 가까운 방향의 벡터가 될 수 있습니다. 이것이 lexical retrieval이 놓친 표현 차이를 semantic retrieval이 찾을 수 있는 이유입니다.

개념 및 API 확인: OpenAI 공식 문서, Embeddings guide와 API reference (접속일: 2026-07-14). 설명과 실습 문장은 AMSL 학습자료용으로 독자적으로 작성했습니다.

기억해야 할 제한

  • embedding은 사실 판정기가 아니라 유사성 계산을 위한 표현입니다.
  • 가까운 문서가 질문에 직접 답하지 않을 수 있습니다.
  • corpus에 정답이 없어도 검색기는 가장 가까운 항목을 반환할 수 있습니다.
  • 다른 embedding model이 만든 벡터는 차원과 공간이 다를 수 있으므로 그대로 섞지 않습니다.
  • model을 바꾸면 기존 문서 index도 같은 model로 다시 만드는 것이 원칙입니다.

3. 작은 2차원 예제로 cosine similarity 보기

실제 embedding은 차원이 크지만, 먼저 두 개의 숫자만 있는 벡터로 계산 흐름을 봅니다.

from math import sqrt


def cosine_similarity(a: list[float], b: list[float]) -> float:
    assert len(a) == len(b)
    dot = sum(x * y for x, y in zip(a, b))
    norm_a = sqrt(sum(x * x for x in a))
    norm_b = sqrt(sum(y * y for y in b))
    if norm_a == 0 or norm_b == 0:
        raise ValueError("zero vector의 cosine similarity는 정의할 수 없습니다.")
    return dot / (norm_a * norm_b)

query = [1.0, 0.0]
same_direction = [0.8, 0.1]
different_direction = [0.1, 0.9]

print(cosine_similarity(query, same_direction))
print(cosine_similarity(query, different_direction))

Cosine similarity는 벡터의 크기보다 방향이 얼마나 비슷한지를 봅니다. 일반적으로 값이 클수록 가깝다고 해석하지만, 점수 범위와 의미는 VectorStore의 distance 전략에 따라 달라질 수 있습니다. 서로 다른 library의 score를 이름만 보고 같은 척도로 비교하지 않습니다.

4. Indexing과 querying을 분리하기

문서와 질문을 embedding하는 시점은 다릅니다.

Indexing

  1. 문서를 load하고 chunk로 나눕니다.
  2. 각 chunk를 embedding합니다.
  3. vector와 page_content, metadata를 함께 저장합니다.

문서가 바뀌지 않았다면 매 질문마다 이 단계를 반복할 필요는 없습니다.

Querying

  1. 새 질문을 같은 embedding model로 embedding합니다.
  2. 저장된 chunk vector와 similarity를 비교합니다.
  3. 상위 k개의 chunk와 metadata를 반환합니다.

질문마다 필요한 것은 query embedding과 검색입니다. 실습에서 같은 문서를 매번 다시 embedding하면 비용과 시간이 불필요하게 늘어납니다.

5. 먼저 offline 모형으로 데이터 흐름 확인하기

제공된 실습은 ConceptEmbedding이라는 결정론적 수업용 모형을 사용합니다. 동의어 일부를 같은 대표 개념으로 바꾸고 hash bucket에 세어 벡터를 만드는 단순한 장치입니다. 실제 embedding model의 언어 이해나 일반화 성능을 흉내 내지 않습니다. 이 모형을 쓰는 목적은 API key, 비용, 네트워크 상태와 관계없이 아래 사항을 먼저 검증하는 것입니다.

  • document와 query가 같은 interface로 vector가 되는가?
  • VectorStore가 본문과 metadata를 함께 반환하는가?
  • 답이 없는 질문에도 nearest result가 반환되는가?
  • 같은 입력으로 다시 실행했을 때 평가 결과가 같은가?

InMemoryVectorStore의 similarity 계산에는 NumPy가 필요합니다. 최신 실습 ZIP의 requirements.txt에 포함되어 있으므로, 안내된 공통 환경을 설치한 뒤 Notebook을 위에서 아래로 실행하면 됩니다.

offline 모형에서 잘 나온 결과는 실제 embedding의 품질을 증명하지 않습니다. 반대로 이 모형의 실패도 실제 model이 반드시 실패한다는 뜻이 아닙니다. Section 8의 평가 방법을 연습하기 위한 통제된 baseline으로만 사용합니다.

6. 선택 실습: 실제 API readiness gate

필수 실습의 ConceptEmbedding 통과는 interface와 데이터 흐름의 재현성만 확인합니다. 실제 embedding model이 한국어 표현 변경을 적절히 검색하는지는 증명하지 않습니다. 이를 혼동하지 않도록 notebook은 다음 두 상태를 따로 출력합니다.

  • offline fixture gate: PASS: 비용 없는 수업용 모형과 metadata 검사가 통과함
  • real API gate: PASS: 실제 text-embedding-3-small로 direct/paraphrase 검색 검사가 통과함

기본 실행에서는 real API gate가 SKIPPED되므로 비용이 발생하지 않습니다. 담당자가 비용 사용을 승인하고 환경변수에 key가 준비된 경우에만 다음처럼 명시적으로 실행합니다.

Jupyter에서 해당 notebook 상단의 API key 설정 셀에 수업용 key를 입력하고 `RUN_LIVE_API = True`로 변경합니다.

공유 notebook이나 shell history에 key를 직접 입력하지 말고, 실제 수업에서는 승인된 secret 주입 방식을 사용합니다. 코드는 key의 존재만 확인하며 값을 출력하거나 저장하지 않습니다. 실행 전에 다음 계획을 먼저 출력합니다.

model: text-embedding-3-small
document inputs: 최대 5
query inputs: 정확히 3 (direct / paraphrase / unanswerable)
total characters: 최대 12,000
planned requests: 4 (문서 batch 1회 + query 3회)
approximate token range: 문자 수 기반 범위이며 과금용 정확한 token 수가 아님

문서는 한 번만 embedding하고 세 query에서 같은 index를 재사용합니다. 응답에 usage가 제공되면 요청별 prompt_tokens와 total_tokens를 출력해 실제 사용량을 남깁니다. 제공되지 않으면 provider usage 기록을 확인합니다. 답이 없는 질문에도 top-1은 반환될 수 있으므로 real gate는 direct/paraphrase의 기대 source만 assert하고, unanswerable 결과는 Section 9의 답변 거부 정책으로 넘깁니다.

2026-07-14에 확인한 OpenAI 공식 가격표에서 text-embedding-3-small은 입력 100만 token당 USD 0.02입니다. 가격은 변경될 수 있으므로 실행 직전에 공식 가격표를 다시 확인합니다. 문자 기반 token 범위는 호출 규모를 사전 점검하기 위한 근사치이며 청구액 계산의 근거로 사용하지 않습니다.

가격 확인: OpenAI 공식 API Pricing (확인일: 2026-07-14). 실제 실행 전에 최신 가격을 다시 확인해야 합니다.

7. Embedding adapter와 in-memory VectorStore

이 Section에서는 별도 서버나 database를 설치하지 않습니다. LangChain의 in-memory VectorStore를 사용해 한 Python process 안에서 vector와 Document를 관리합니다. process를 종료하면 index가 사라지므로 제품용 영구 저장소가 아닙니다.

구조는 다음과 같습니다.

from openai import OpenAI
from langchain_core.embeddings import Embeddings
from langchain_core.vectorstores import InMemoryVectorStore


class OpenAIEmbeddingAdapter(Embeddings):
    def __init__(self, client: OpenAI, model: str):
        self.client = client
        self.model = model

    def embed_documents(self, texts: list[str]) -> list[list[float]]:
        response = self.client.embeddings.create(model=self.model, input=texts)
        return [item.embedding for item in response.data]

    def embed_query(self, text: str) -> list[float]:
        response = self.client.embeddings.create(model=self.model, input=text)
        return response.data[0].embedding

adapter는 OpenAI API의 embedding 결과를 LangChain Embeddings interface에 맞춥니다. adapter를 작성했다고 embedding algorithm을 직접 구현한 것은 아닙니다. 실제 벡터는 API의 embedding model이 만듭니다.

embeddings = OpenAIEmbeddingAdapter(
    client=OpenAI(),
    model="text-embedding-3-small",
)
vector_store = InMemoryVectorStore(embedding=embeddings)
vector_store.add_documents(chunks)

모델명은 제작 시점의 공식 문서와 수업 계정 지원 여부를 다시 확인한 뒤 고정합니다.

Interface 확인: LangChain 공식 API reference의 InMemoryVectorStore 및 Embeddings interface (접속일: 2026-07-14). Adapter와 corpus는 독자적으로 작성했습니다.

8. 필수 실습 A: lexical과 vector 검색 비교

Section 6의 corpus를 사용해 다음 세 종류의 질문을 준비합니다.

  1. 단어가 직접 겹치는 질문: “API key를 어디에 올리면 안 되나요?”
  2. 표현을 바꾼 질문: “비밀 인증정보가 유출된 것 같으면 어떻게 대응하나요?”
  3. 답이 없는 질문: “연구실 정기 회의는 매주 몇 시인가요?”

각 질문에 대해 lexical 검색과 vector 검색의 top 3를 다음 형태로 출력합니다.

[vector rank 1]
source_id: ...
chunk_id: ...
score: ...
preview: ...

비교할 때 모델 답변은 만들지 않습니다. 이번 실습의 대상은 retrieved chunks입니다.

  • 같은 주제지만 질문에 답하지 않는 chunk가 높은 순위에 있나요?
  • 답이 없는 질문에서도 top 1이 반환되나요?
  • Section 6의 small/large chunk 설정 중 어느 쪽이 더 적절한 근거를 반환했나요?

9. 필수 실습 B: model과 index의 일관성

다음 metadata를 index 단위로 기록합니다.

index_config = {
    "embedding_model": "text-embedding-3-small",
    "source_version": "section06-v1",
    "split_setting": "small",
}

query model이 index_config["embedding_model"]과 다르면 검색을 실행하기 전에 오류를 냅니다. 실제 저장 시스템에서는 model명 외에도 vector 차원, 전처리 방식, corpus version을 함께 관리해야 합니다.

10. 높은 similarity가 정답을 보장하지 않는 이유

답이 없는 회의 시간 질문에도 검색기는 가장 가까운 chunk를 반환할 수 있습니다. 이때 높은 순위는 “전체 후보 중 상대적으로 가깝다”는 뜻이지 “이 문서에 정답이 있다”는 보장이 아닙니다.

또한 score threshold 하나만으로 답변 가능 여부를 완벽히 결정할 수 없습니다. 질문 종류와 corpus에 따라 점수 분포가 달라지고, embedding model이나 distance 함수가 바뀌면 점수 해석도 달라집니다. Section 8에서 기대 source가 있는 평가 질문 세트를 만들고 top-k와 threshold를 실제 결과로 조정합니다.

11. 실패 진단 순서

  1. 질문에 답하는 내용이 corpus에 실제로 있는지 확인합니다.
  2. 해당 내용이 loader와 splitter를 거쳐 온전히 남았는지 확인합니다.
  3. 문서와 질문이 같은 embedding model로 처리되었는지 확인합니다.
  4. 검색된 chunk의 본문과 metadata를 직접 봅니다.
  5. top-k와 filter를 확인합니다.
  6. 그 다음 embedding model 변경을 검토합니다.

12. 결과 확인

  • embedding과 similarity의 관계를 자기 말로 설명했습니다.
  • cosine similarity dry run을 API 없이 실행했습니다.
  • index 단계와 query 단계를 구분했습니다.
  • Section 6 chunk의 metadata가 검색 결과에도 유지되는지 확인했습니다.
  • 세 종류의 질문에서 lexical과 vector 검색 결과를 비교했습니다.
  • 답이 없는 질문에도 nearest result가 나올 수 있음을 확인했습니다.
  • similarity와 correctness가 다르다는 사례를 하나 기록했습니다.
