Section 8. Retriever를 설계하고 검색 품질 점검하기

예상 시간: 필수 125~150분 / 선택 심화 30~45분
선수 과정: Section 6~7
API 비용: 필수 실습은 API 호출 없음

이번 Section의 위치

Section 7에서는 같은 corpus를 lexical 검색과 vector 검색으로 찾아보았습니다. 하지만 몇 개의 결과를 눈으로 보고 “vector 검색이 더 자연스럽다”고 말하는 것만으로는 검색기가 좋아졌다고 판단할 수 없습니다. 질문을 바꾸면 결론이 달라질 수 있고, top result가 그럴듯해도 실제 정답 source를 놓칠 수 있습니다.

이번 Section에서는 생성 모델을 호출하지 않습니다. 질문을 받아 Document 목록을 반환하는 retriever의 경계를 분명히 하고, 작은 정답 세트로 검색 설정을 측정합니다.

핵심 질문

검색 결과가 있다는 사실과 질문에 필요한 근거를 찾았다는 사실을 어떻게 구분할까요?

학습 목표

  1. VectorStore와 Retriever의 책임을 구분합니다.
  2. k, score threshold, metadata filter가 후보 결과에 미치는 영향을 실험합니다.
  3. 8~12개의 평가 질문에 기대 source와 answerability를 기록합니다.
  4. Hit@k와 Recall@k를 계산하고 수치가 의미하는 범위를 설명합니다.
  5. 실패를 query, chunk, corpus, configuration, representation 문제로 분류합니다.
  6. 답이 없는 질문을 일반적인 검색 hit와 별도로 평가합니다.

이번 Section의 필수선과 발전선

  • 필수선: 제공된 평가 질문으로 k=1과 k=3을 비교하고, 기대 source가 검색되었는지 질문별로 기록합니다.
  • 발전선: 질문 세트 확장과 MultiQuery/ensemble 비교는 선택입니다.
  • 복습 신호: 비율 계산이 익숙하지 않다면 4개 질문 중 3개 적중을 3/4로 손으로 계산한 뒤 코드 결과와 맞춥니다.

처음부터 완벽한 gold set을 만드는 것은 요구하지 않습니다. 제공된 기대 source를 검토하고 학습자가 직접 추가하는 질문은 2개면 충분합니다. 라벨이 애매하면 임의로 확정하지 말고 needs_review로 표시해 함께 검토합니다.

1. 선수지식 확인

  • Section 7의 index 단계와 query 단계를 한 문장씩 설명할 수 있나요?
  • k=3은 검색기가 세 문서 모두 옳다고 판정했다는 뜻일까요?
  • 기대 source가 없는 질문을 일반 Recall 계산에 넣으면 어떤 문제가 생길까요?
  • metadata filter를 잘못 적용하면 embedding이 좋아도 정답을 찾을 수 있을까요?

2. VectorStore와 Retriever의 책임 경계

VectorStore는 vector, 본문과 metadata를 저장하고 similarity search 같은 연산을 제공합니다. Retriever는 사용자 질문을 받아 downstream 단계가 사용할 Document 목록을 반환하는 인터페이스입니다. Retriever 안에서는 VectorStore 검색뿐 아니라 filter, threshold, query 변환 또는 여러 검색 결과의 결합이 일어날 수 있습니다.

question
   ↓
Retriever policy: k, filter, threshold
   ↓
VectorStore search
   ↓
ranked Documents + metadata

둘을 구분하면 “vector가 잘못 저장되었는가?”와 “retriever 정책이 좋은 후보를 제거했는가?”를 따로 확인할 수 있습니다. 이번 실습의 retriever는 작은 함수이지만 입력과 출력의 계약은 분명히 유지합니다.

def retrieve(question: str, k: int) -> list[tuple[Document, float]]:
    return store.similarity_search_with_score(question, k=k)
인터페이스와 검색 흐름 확인: LangChain 공식 문서, Retrieval 및 Vector stores (접속일: 2026-07-14). 설명과 예제는 독자적으로 작성했습니다.

3. Gold evaluation set 만들기

검색 설정을 비교하려면 설정을 보기 전에 질문과 기대값을 먼저 고정합니다. 그렇지 않으면 현재 검색기가 잘 찾는 질문만 남기는 방식으로 평가가 왜곡될 수 있습니다.

각 사례에는 최소한 다음 필드를 둡니다.

{
    "id": "q01",
    "question": "API key를 공개 저장소에 올려도 되나요?",
    "expected_source": "security-guide",
}

corpus로 답할 수 없는 질문은 expected_source=None으로 표시합니다. 정답 source가 여러 개라면 문자열 하나로 억지로 줄이지 않고 expected_sources 집합과 채점 규칙을 별도로 정해야 합니다. 이번 corpus는 metric 계산을 명확히 보기 위해 답할 수 있는 질문마다 대표 source 하나를 지정했습니다.

질문 구성 원칙

  • 문서의 표현을 그대로 포함한 질문과 바꿔 쓴 질문을 섞습니다.
  • 다섯 category가 한쪽으로 치우치지 않게 합니다.
  • 답할 수 없는 질문을 최소 두 개 넣습니다.
  • 개인정보나 실제 연구실 내부 정보를 정답 데이터에 넣지 않습니다.
  • 질문을 만든 사람 외의 검토자가 기대 source를 원문과 대조합니다.

4. Hit@k와 Recall@k

한 질문에서 기대 source가 top-k 안에 하나라도 있으면 hit로 셉니다. 답할 수 있는 질문이 N개일 때 이번 실습의 Recall@k는 다음과 같이 계산합니다.

Recall@k = top-k에 기대 source가 포함된 질문 수 / 답할 수 있는 전체 질문 수

대표 source가 하나뿐인 현재 구성에서는 평균 Hit@k와 같은 값입니다. 여러 관련 문서를 모두 찾아야 하는 평가에서는 질문별 relevant 문서 중 몇 개를 찾았는지 계산해야 하므로 같은 방식으로 단순화하면 안 됩니다.

k가 커지면 Recall@k는 보통 유지되거나 올라갑니다. 그러나 LLM에게 전달되는 무관한 문서, 입력 길이와 비용도 늘 수 있습니다. 따라서 Recall만 보고 가장 큰 k를 선택하지 않습니다. Section 9에서는 context에 들어간 근거의 정확성까지 함께 봅니다.

평가 개념 확인: Manning, Raghavan, Schütze, Introduction to Information Retrieval, Evaluation in information retrieval (접속일: 2026-07-14). 수업용 계산 예시는 독자적으로 작성했습니다.

5. k=1과 k=3 비교

제공된 12개 질문 중 답할 수 있는 10개에 대해 두 설정을 실행합니다.

for k in (1, 3):
    print(f"Recall@{k}: {recall_at_k(k):.3f}")

제작용 offline baseline에서는 Recall@1이 0.900, Recall@3이 1.000으로 재현되었습니다. 이 수치는 실제 OpenAI embedding의 예상 성능이 아니라, metric 코드와 실패 분석 흐름이 작동하는지 확인한 fixture 결과입니다. corpus, chunk, model 또는 질문이 바뀌면 baseline을 새로 저장합니다.

관찰할 내용:

  • top 1에서 놓쳤지만 top 3에서 찾은 질문은 무엇인가요?
  • top 3에 추가된 두 문서는 질문과 실제로 관련이 있나요?
  • k=3이 generation 단계에 더 좋은 선택이라고 바로 결론 내릴 수 있나요?

6. Metadata filter

filter는 특정 category, 날짜, 접근 권한 또는 문서 종류로 검색 공간을 제한합니다. embedding이 구분하기 어려운 영역을 명시적 조건으로 좁힐 수 있지만, metadata가 틀리거나 filter를 잘못 고르면 정답 문서는 검색 전에 제거됩니다.

filter_fn = lambda doc: doc.metadata.get("category") == "development"
results = store.similarity_search_with_score(
    "에러 질문에는 어떤 정보를 포함하나요?",
    k=3,
    filter=filter_fn,
)

필수 실습에서는 같은 질문에 filter 없음, 올바른 development, 잘못된 security를 적용합니다. filter 결과가 기대와 다르다면 embedding model을 바꾸기 전에 원본 metadata와 filter callable이 받는 객체를 확인합니다.

7. Score threshold

top-k 검색은 corpus에 답이 없어도 가장 가까운 후보를 반환합니다. threshold는 점수가 낮은 결과를 거부하는 한 가지 정책이지만 보편적인 정답값은 없습니다.

제작 fixture에서 0.30을 시작점으로 적용하면 답 없음 두 질문은 모두 거부하지만, 답할 수 있는 질문 일부도 거부합니다. 즉 false acceptance를 줄이는 대신 false rejection이 늘어납니다. 이 결과가 threshold를 평가 세트로 조정해야 하는 이유입니다.

다음 값을 함께 기록합니다.

  • answerable 질문 중 accepted 비율
  • 기대 source가 top 1이고 accepted인 비율
  • unanswerable 질문 중 잘못 accepted된 비율
  • threshold 바로 위와 아래에 있는 실제 질문

score의 범위와 방향은 store 구현과 distance 전략에 따라 달라질 수 있습니다. 다른 backend로 옮길 때 0.30을 그대로 복사하지 않습니다.

8. 답이 없는 질문은 별도로 평가하기

expected_source=None인 질문은 일반 Recall의 분모에서 제외합니다. 기대 source가 없기 때문에 “찾지 못함”이 올바른 동작일 수 있기 때문입니다. 대신 retriever가 안전하게 빈 결과를 반환하거나 not_answered 후보 상태로 보냈는지 봅니다.

현재 단계에서는 threshold가 검색 결과를 거부할 수 있게 만드는 데까지만 다룹니다. threshold를 넘었다고 답변 가능성이 확정되는 것은 아닙니다. Section 9에서는 retrieved context 안에 실제 답 근거가 있는지 validation을 한 번 더 수행합니다.

9. 실패 분류

검색 miss를 발견하면 아래 범주 중 하나로 임시 분류하고 증거를 기록합니다.

  • query: 표현이 모호하거나 여러 의도를 가짐
  • chunk: 필요한 문장이 경계에서 분리되거나 주변 문맥이 부족함
  • corpus: 답이 원문에 없거나 source가 누락됨
  • configuration: k, filter, threshold가 좋은 후보를 제거함
  • representation: embedding이 필요한 의미 관계를 충분히 표현하지 못함
  • label: 기대 source 자체가 틀리거나 복수 정답을 하나로 제한함

한 실패에 원인이 여러 개일 수 있습니다. 처음부터 “embedding 문제”로 단정하지 말고 원문 존재 → chunk → metadata → 설정 → representation → label 순서로 확인합니다.

10. 필수 실습 순서

  1. 제공된 corpus 5개와 12개 평가 질문을 읽고 기대 source를 원문과 대조합니다.
  2. API 없는 ConceptEmbedding으로 index를 생성합니다.
  3. k=1, k=3의 Recall을 계산합니다.
  4. 한 질문에 올바른 filter와 잘못된 filter를 각각 적용합니다.
  5. threshold 0.30의 acceptance를 answerable/unanswerable로 나누어 봅니다.
  6. top-1 miss와 threshold false rejection을 최소 두 개 분류합니다.
  7. 실제 embedding 선택 실습을 한다면 같은 평가 파일을 그대로 실행하고 별도 결과로 저장합니다.

실행 명령:

uv run --python 3.11 --with-requirements requirements.txt --with numpy==2.4.3 \

11. 실패를 디버깅하는 순서

  1. 평가 질문의 기대 source가 원문에 실제로 답을 포함하는지 확인합니다.
  2. 기대 source의 chunk가 index에 있고 metadata가 유지되는지 확인합니다.
  3. filter 전 결과와 filter 후 결과를 비교합니다.
  4. top-k 전체 source와 score를 출력합니다.
  5. threshold를 적용하기 전 raw 결과를 확인합니다.
  6. 같은 model/index config인지 확인합니다.
  7. 마지막으로 query 수정, chunk 변경 또는 embedding 변경 실험을 하나씩 수행합니다.

설정을 여러 개 동시에 바꾸면 어느 변경이 결과에 영향을 주었는지 알 수 없습니다.

12. 결과 확인

  • VectorStore와 Retriever의 역할을 입력·출력 기준으로 구분했습니다.
  • 평가 설정을 보기 전에 12개 질문과 기대 source를 확인했습니다.
  • Recall@1과 Recall@3를 직접 계산하고 차이를 설명했습니다.
  • 올바른 filter가 검색 공간을 좁히고, 잘못된 filter가 정답을 제거하는 사례를 실행했습니다.
  • 답 없음 질문을 Recall과 분리해 threshold 정책으로 평가했습니다.
  • 최소 두 실패를 원인 범주와 근거를 포함해 기록했습니다.
  • 생성 모델을 호출하지 않고 retrieval 실패를 진단했습니다.

13. 짧은 제출물

  1. Recall@1, Recall@3와 각 설정의 장단점
  2. filter 전/후 top source 비교
  3. 답 없음 질문 두 개의 top score와 acceptance
  4. 실패 사례 두 개: 예상, 실제, 원인 범주, 다음 한 가지 실험
  5. Section 9에서 사용할 k, filter, threshold 후보와 선택 근거

14. 선택 심화

  • 질문별 relevant source가 여러 개인 평가 세트로 precision과 recall을 따로 계산
  • lexical+vector ensemble을 동일 질문 세트에서 baseline과 비교
  • MultiQueryRetriever는 단순 query 실패가 확인된 사례에만 적용
  • threshold별 false acceptance/false rejection curve 작성

고급 retriever를 추가하기 전에 단순 baseline의 실패가 무엇인지 먼저 기록합니다. 새 구성요소가 metric 또는 명확한 실패 사례를 개선하지 못하면 필수 pipeline에 넣지 않습니다.
