Section 6. 긴 문서를 검색 가능한 단위로 만들기

예상 시간: 필수 110~140분 / 선택 심화 30~45분
선수 과정: Section 0~4
API 비용: 필수 실습은 API 호출 없음

이 Section을 시작하기 전에

Section 3에서는 짧은 문서 몇 개를 직접 준비하고, 질문에 포함된 단어와 문서의 단어가 얼마나 겹치는지 비교했습니다. 이 방식은 RAG의 검색 → 근거 → 답변 흐름을 확인하기에는 충분하지만, 실제 PDF나 보고서는 길고 구조도 일정하지 않습니다.

이번 Section에서는 아직 embedding이나 vector database를 사용하지 않습니다. 먼저 긴 문서를 작은 단위로 나누고, 각 조각이 원문의 어느 위치에서 왔는지 잃어버리지 않는 방법을 다룹니다. 검색 모델을 바꾸더라도 문서 준비 과정이 잘못되어 있으면 좋은 근거를 찾기 어렵기 때문입니다.

핵심 질문

긴 문서를 왜 나누어야 하며, 나눈 뒤에도 원문의 위치와 의미를 어떻게 보존할 수 있을까요?

학습 목표

이 Section을 마치면 다음을 할 수 있어야 합니다.

  1. 문서 수집(loading), 분할(splitting), 검색(retrieval)을 서로 다른 단계로 구분합니다.
  2. LangChain Document의 page_content와 metadata가 맡는 역할을 설명합니다.
  3. 긴 문서를 RecursiveCharacterTextSplitter로 나누고 설정에 따른 차이를 비교합니다.
  4. 모든 chunk에 source_id, chunk_id, 원문 위치를 유지합니다.
  5. chunk가 너무 작거나 클 때 생기는 검색 실패를 사례로 설명합니다.

이번 Section의 필수선과 발전선

  • 필수선: 제공된 텍스트를 두 설정으로 분할하고, source_id·chunk_id·원문 위치가 보존되는지 확인합니다.
  • 발전선: PDF/Markdown loader 비교, token 기준 분할, semantic chunking은 결과를 빨리 낸 학습자만 선택합니다.
  • 복습 신호: Python dictionary에서 값을 읽거나 assert 실패 메시지를 해석하기 어렵다면 실습 전에 해당 두 항목을 15분간 복습합니다.

이 장에서 loader 종류를 많이 외울 필요는 없습니다. 이후 장에서 계속 사용하는 핵심은 “본문과 추적 정보를 분리해 보존한다”와 “분할 설정은 검색 결과로 검증한다”는 두 원칙입니다.

1. 먼저 확인하기

아래 질문에 완벽히 답하지 못해도 괜찮습니다. 모르는 항목은 본문을 읽은 뒤 다시 확인합니다.

  • Python의 문자열과 dictionary가 무엇인지 설명할 수 있나요?
  • 문서 본문과 파일명은 왜 서로 다른 필드로 관리하는 편이 좋을까요?
  • 같은 원문에서 잘라낸 두 문자열을 나중에 다시 구분하려면 어떤 정보가 필요할까요?
  • 검색 결과가 틀렸을 때 “검색 알고리즘” 외에 어떤 원인이 있을까요?
개념 확인: LangChain 공식 문서, Document loader integrations (접속일: 2026-07-14). 이 페이지의 설명과 예제는 독자적으로 작성했습니다.

2. 문서를 모델에 넣기 전에 일어나는 일

RAG는 흔히 “관련 문서를 검색해서 LLM에게 준다”라고 설명합니다. 하지만 실제로 검색하기 전에도 여러 단계가 있습니다.

  1. Load: 파일, 웹 페이지 또는 데이터베이스에서 내용을 읽습니다.
  2. Normalize: 본문과 출처 정보를 일정한 데이터 구조로 맞춥니다.
  3. Split: 긴 본문을 검색 가능한 크기의 조각으로 나눕니다.
  4. Index: 각 조각을 검색 시스템이 찾을 수 있도록 저장합니다.
  5. Retrieve: 질문에 관련된 조각을 찾습니다.
  6. Generate: 찾은 조각을 근거로 답변을 만듭니다.

이번 Section은 1~3단계에 집중합니다. Section 7에서 index와 semantic retrieval을 연결합니다.

단계를 나누는 이유는 실패 원인을 찾기 위해서입니다. PDF에서 표가 누락되었다면 loader 문제일 수 있고, 문장의 앞뒤가 서로 다른 chunk로 잘렸다면 splitter 문제일 수 있습니다. 이 문제를 embedding model이나 prompt만 바꾸어 해결하려 하면 원인을 놓치게 됩니다.

3. Document: 본문과 출처를 함께 운반하기

LangChain의 Document는 크게 두 부분을 가집니다.

  • page_content: 검색하고 모델에게 전달할 본문
  • metadata: 파일명, 페이지, 문서 ID처럼 본문을 추적하는 정보
from langchain_core.documents import Document

document = Document(
    page_content="API key는 공개 저장소에 올리지 않습니다.",
    metadata={
        "source_id": "onboarding-guide",
        "title": "연구실 온보딩 가이드",
        "page": 3,
    },
)

metadata는 답변을 그럴듯하게 꾸미기 위한 장식이 아닙니다. 검색 결과가 어느 문서에서 왔는지 확인하고, 잘못된 답변을 원문과 대조하며, 사용자에게 출처를 표시하기 위한 데이터입니다.

metadata의 최소 계약

이번 과정에서는 모든 원본 문서에 다음 필드를 둡니다.

source_id   사람이 읽을 수 있고 문서마다 고유한 ID
title       화면에 표시할 문서 제목
source_type txt, pdf, web 등 원본 형태

분할된 chunk에는 다음 필드를 추가합니다.

chunk_id    같은 source 안에서 고유한 조각 ID
char_start  원문에서 시작한 문자 위치
char_end    원문에서 끝난 문자 위치

PDF loader가 신뢰할 수 있는 page 번호를 제공한다면 page도 유지합니다. 다만 loader마다 페이지 번호가 0부터 시작하는지 1부터 시작하는지 다를 수 있으므로 화면에 표시하기 전에 확인해야 합니다.

4. 왜 긴 문서를 나누나요?

긴 문서를 통째로 넣는 방식에는 세 가지 문제가 있습니다.

  1. 질문과 관계없는 내용까지 함께 전달되어 비용과 입력 길이가 늘어납니다.
  2. 검색 결과가 “문서 전체”라면 실제 근거가 어디에 있는지 찾기 어렵습니다.
  3. 관련 문장이 짧은데 주변의 무관한 내용이 많으면 검색 점수가 흐려질 수 있습니다.

반대로 너무 잘게 나누어도 문제가 생깁니다.

원문: API key는 공개 저장소에 올리지 않습니다.

잘못된 분할 A: "API key는 공개"
잘못된 분할 B: "저장소에 올리지 않습니다."

각 조각만 보면 금지 규칙의 목적어와 행동이 분리됩니다. 검색은 A만 찾고 답변은 “공개해도 된다”로 잘못 해석할 수도 있습니다. 좋은 chunk는 무조건 짧은 조각이 아니라, 질문에 답할 수 있는 최소한의 의미 단위를 보존한 조각입니다.

5. 세 가지 설정

텍스트 분할에서 먼저 볼 값은 다음과 같습니다.

  • chunk_size: 한 chunk가 가질 목표 최대 크기
  • chunk_overlap: 인접 chunk가 반복해서 포함할 범위
  • length_function: 길이를 문자, token 등 무엇으로 셀지 결정하는 함수

overlap은 경계에서 문맥이 끊기는 문제를 줄이지만, 같은 문장이 여러 chunk에 중복되어 검색 결과가 단조로워질 수 있습니다. 따라서 overlap을 크게 설정한다고 항상 좋아지지는 않습니다.

API 및 동작 확인: LangChain 공식 문서, Recursive text splitter (접속일: 2026-07-14). 예제 문서와 비교 실험은 AMSL 학습자료용으로 새로 작성했습니다.

RecursiveCharacterTextSplitter

이 splitter는 먼저 큰 경계에서 나누고, 조각이 여전히 크면 더 작은 경계를 시도합니다. 일반 텍스트에서는 보통 문단, 줄바꿈, 공백, 문자 순으로 경계를 좁혀가는 방식으로 사용할 수 있습니다.

from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=300,
    chunk_overlap=50,
    add_start_index=True,
    keep_separator="end",
)

chunks = splitter.split_documents([document])

여기서 300은 “좋은 기본값”이 아닙니다. 실습용 짧은 한국어 문서에서 차이를 관찰하기 위한 시작점입니다. 실제 설정은 문서 구조, 질문의 크기, embedding model의 입력 한도, 검색 평가 결과를 함께 보고 정합니다. keep_separator="end"는 문장 구분 기호를 앞 chunk의 끝에 남겨 다음 chunk가 마침표로 시작하는 현상을 줄입니다. 하지만 문서의 문장 부호와 줄바꿈이 일관되지 않다면 이것만으로 자연스러운 경계를 보장하지는 않습니다.

6. 필수 실습 A: 분할 결과 관찰하기

실습에서는 약 1,500~2,500자 분량의 가상 연구실 온보딩 문서를 사용합니다. 실제 연구실의 비공개 규정이나 개인정보는 포함하지 않습니다.

다음 두 설정을 비교합니다.

settings = {
    "small": {"chunk_size": 180, "chunk_overlap": 20},
    "large": {"chunk_size": 500, "chunk_overlap": 80},
}

각 결과에서 아래 항목을 출력합니다.

  • chunk 개수
  • 최소/평균/최대 길이
  • 각 chunk의 앞 80자
  • 원문 시작 위치
  • 문장이 부자연스럽게 끊긴 사례
  • 같은 문장이 overlap으로 반복된 사례

7. 필수 실습 B: 원문 위치 검증하기

분할된 chunk마다 다음 조건을 검사합니다.

assert chunk.metadata["source_id"] == "onboarding-guide"
assert "chunk_id" in chunk.metadata
assert "char_start" in chunk.metadata
assert "char_end" in chunk.metadata

그 다음 char_start, char_end로 원문을 다시 잘라 page_content와 비교합니다. overlap 처리나 공백 정규화 때문에 완전히 같지 않다면 그 이유를 기록합니다. 단순히 assertion을 없애서 통과시키지 않습니다.

중요한 실패 사례

다음 중 하나가 발생하도록 일부러 잘못된 처리도 실행합니다.

  • 분할 후 metadata를 새 dictionary로 교체해 source_id가 사라짐
  • 서로 다른 문서가 같은 chunk_id를 사용함
  • 공백을 정리한 뒤 이전 문자 위치를 그대로 사용함

오류를 본 뒤, 검색 결과에서 원문 출처를 복구할 수 있는지 확인합니다.

8. 문자 수와 token 수는 같지 않습니다

len(text)는 Python 문자열의 문자 수를 세지만, LLM과 embedding model은 token 단위로 입력을 처리합니다. 한국어, 영어, 숫자, 기호가 섞이면 같은 문자 수라도 token 수가 다를 수 있습니다.

이번 필수 실습에서는 문자 기준 splitter를 사용해 문서 분할의 원리를 먼저 봅니다. 선택 심화에서는 사용 중인 모델에 맞는 tokenizer를 이용해 같은 문서의 문자 수와 token 수를 비교합니다. 특정 모델 tokenizer 결과를 다른 모델에도 그대로 적용할 수 있다고 가정하지 않습니다.

9. 실패를 진단하는 순서

나중에 검색 결과가 좋지 않을 때 아래 순서로 확인합니다.

  1. 원문이 loader에서 빠지거나 깨지지 않았는가?
  2. 질문에 답하는 문장이 corpus에 실제로 존재하는가?
  3. 필요한 문장이 chunk 경계에서 분리되지 않았는가?
  4. metadata가 분할 후에도 유지되었는가?
  5. 그 다음에 retrieval 설정과 embedding을 확인합니다.

처음부터 embedding model이나 prompt를 바꾸면 문서 준비 단계의 오류를 가릴 수 있습니다.

10. 결과 확인

아래 조건을 모두 만족하면 필수 과정을 마친 것입니다.

  • loading, splitting, indexing, retrieval을 각각 한 문장으로 구분할 수 있습니다.
  • 두 가지 chunk 설정의 결과를 수치와 실제 경계 사례로 비교했습니다.
  • 모든 chunk에서 source_id와 고유한 chunk_id를 확인했습니다.
  • 하나 이상의 chunk에서 원문 위치를 되찾아 검증했습니다.
  • 작은 chunk와 큰 chunk가 각각 실패할 수 있는 사례를 설명했습니다.
  • API 호출 없이 실습 전체를 실행했습니다.

12. 선택 심화

  • token 기준 분할과 문자 기준 분할 비교
  • Markdown 제목을 metadata로 유지하는 분할
  • 표가 포함된 PDF에서 loader별 추출 결과 비교
  • 의미 기반 chunking은 개념만 확인하고, 필수 과정에는 포함하지 않음
