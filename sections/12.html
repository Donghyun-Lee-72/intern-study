Section 12. 미니 프로젝트: 근거 기반 연구실 도우미

예상 시간: 설계 90분 + 구현 4~7시간 + 평가·회고 2시간
선수 과정: Section 6~11
API 비용: 설계·평가기 self-test는 없음. 실제 model 사용 시 팀별 예산 상한을 먼저 정함

프로젝트의 목적

마지막 Section은 기능을 더 배우는 장이 아닙니다. 앞서 만든 ingestion, retrieval, evidence validation, 평가, workflow/agent 제어를 작은 시스템 하나로 묶고 잘 되는 질문뿐 아니라 실패와 거부도 증명합니다. UI나 배포보다 재현 가능한 동작과 평가 보고서가 우선입니다.

핵심 질문

작은 근거 기반 시스템이 답할 수 있는 범위와 답하지 않아야 할 범위를 어떻게 측정하고 설명할까요?

학습 목표

  1. 문제, 사용자, corpus, 답변 범위를 한 문단으로 제한합니다.
  2. source와 chunk를 추적하는 ingestion/retrieval pipeline을 구현합니다.
  3. 답변·답 없음·사람 검토 상태를 구분하고 retrieval·인용 무결성·의미적 지지를 나누어 검증합니다.
  4. 10개 이상의 질문으로 retrieval, 답변, 정책을 분리 평가합니다.
  5. 알려진 한계와 다음 개선 순서를 근거와 함께 제시합니다.

프로젝트 운영 방식과 수준별 범위

이 프로젝트는 한 번에 끝내는 개인 과제가 아니라 네 체크포인트로 나누어 진행합니다. 기본안은 3명이 하나의 corpus와 평가 세트를 공유하고, 각자 한 구성요소의 개선안을 맡는 팀 프로젝트입니다. 개인별 이해도는 자신이 맡지 않은 단계도 입출력과 실패 원인을 설명하는 짧은 구두 점검으로 확인합니다.

  • 공통 필수선: 통제된 corpus, retrieval baseline, 근거 검증, 답 없음/검토 정책, baseline 대비 한 가지 변경
  • 역할 예시: 문서·chunk 담당 / retrieval·평가 담당 / answer·workflow 안전장치 담당
  • 개인 발전선: 실제 embedding·model 연결, LangGraph 전환, UI 중 하나만 선택
  • 범위 축소 기준: 체크포인트 B에서 baseline이 재현되지 않으면 UI와 Agent를 제외하고 RAG 평가까지 완성합니다.

구현 속도가 다른 것은 감점 사유가 아닙니다. 다만 한 사람이 전체 코드를 작성하고 나머지가 실행만 하는 방식은 피합니다. 각자 변경한 한 가지와 그 변경 전후 결과를 남깁니다.

1. 시작 전 결정할 네 문장

구현 전에 다음을 채웁니다.

사용자:
도와줄 업무 한 가지:
사용할 수 있는 문서:
명시적으로 하지 않을 일:

“연구실의 모든 일을 돕는다”는 범위가 아닙니다. 예를 들어 “신규 인턴이 공개 온보딩 문서에서 환경 설정과 보안 규칙을 찾도록 돕는다” 정도로 좁힙니다.

2. 세 가지 프로젝트 예시

  • 공개·자체 작성 온보딩 문서 QA
  • 재배포가 허용된 논문 초록에서 방법론 위치 찾기
  • 비식별화한 가상 회의록에서 action item 검색

실제 개인정보, 비공개 연구자료, 재배포 권한이 불명확한 PDF는 제출 corpus에 넣지 않습니다. 문서 URL이 공개되어 있다는 사실만으로 복제·재배포 권한이 생기지는 않습니다.

3. 최소 기능선

다음 항목은 모두 필요합니다.

  1. 5~20개의 통제된 문서와 문서별 source_id, 제목, 이용 근거
  2. ingestion과 index 생성 코드
  3. 검색 또는 답변 검증을 포함한 최소 두 tool/node
  4. answered, not_answered, needs_review 상태
  5. 검색된 원문에 실제 존재하는 quote와 source 표시, 주요 주장별 의미적 지지 검토
  6. 정상·답 없음·위험 요청을 포함한 10개 이상의 평가 질문
  7. 실행 방법, 환경변수, 비용 상한을 적은 README
  8. 실패 분석과 알려진 한계

웹 UI, 대규모 DB, 멀티모달, multi-agent는 평가가 끝난 뒤의 선택 사항입니다.

4. Architecture와 책임 경계

documents → normalize/split → index
question → policy check → retrieve → answer or refuse → evidence validation → response
                                              ↘ needs_review
evaluation cases ─────────────────────────────→ evaluator/report

각 단계의 입력과 출력을 README에 씁니다. Retriever가 답변을 만들거나, generator가 source ID를 새로 만들어내게 하지 않습니다. 평가 코드는 답변 생성 코드와 분리합니다.

5. 데이터 카드

Corpus와 함께 다음을 기록합니다.

  • 문서 수, 언어, 형식, 수집/작성일
  • 소유자 또는 라이선스와 수업 내 이용 근거
  • 개인정보·비밀정보 제거 방법
  • 예상 질문과 포함하지 않은 범위
  • 업데이트 시 index를 다시 만드는 절차

Source ID는 파일 경로나 임시 순번에만 의존하지 않고 버전이 바뀌어도 추적 가능한 이름을 사용합니다.

6. Starter와 먼저 테스트할 평가기

notebooks/12_capstone_starter.ipynb에는 완성 retrieval이나 answer 함수가 없습니다. 대신 다음 계약을 제공합니다.

  • Evidence(source_id, chunk_id, quote)
  • Answer(status, text, evidence)
  • EvaluationCase(question, answerability, expected_source_ids)
  • 원문에 quote가 존재하는지 확인하는 validate_evidence
  • retrieval log의 기대 source와 답변의 exact quote integrity를 분리하는 score_case
  • 의미적 지지를 자동으로 꾸며내지 않고 사람 검토를 결합하는 ClaimSupportReview

구현보다 먼저 평가기 self-test를 실행합니다. 조작된 quote와 위험 요청에 답한 사례를 평가기가 잡지 못하면 시스템 구현을 진행하지 않습니다. 또한 원문에 정확히 있는 quote지만 답변 주장을 지지하지 않는 fixture를 둡니다. 이 사례는 substring 검사를 통과해야 하고, 별도의 의미 검토에서 실패해야 올바른 평가기입니다.

7. 평가 질문 설계

최소 10개 질문을 다음처럼 구성합니다.

  • 직접 답할 수 있는 질문 4개 이상
  • 표현을 바꿔 답할 수 있는 질문 2개 이상
  • corpus에 답이 없는 질문 2개 이상
  • prompt injection, secret 요청, 범위 밖 변경 같은 위험 질문 2개 이상

실제 개발 중에 본 질문만 평가에 쓰면 과적합될 수 있습니다. 질문 일부는 corpus를 작성하지 않은 사람이 만들거나, 최소한 개발용과 최종 확인용으로 나눕니다.

8. 측정 항목

측정은 다음 세 층을 합치지 않습니다.

  1. Source retrieval: 기대 source/chunk가 검색되었는지
  2. Exact quote integrity: 제시된 quote가 검색된 원문에 정확히 존재하는지
  3. Semantic claim support: quote가 답변의 주요 주장을 실제로 지지하는지

1과 2는 고정 ID와 문자열로 자동 확인할 수 있습니다. 3은 단순 문자열 포함 여부로 증명하지 않습니다. 주요 주장별로 supported / contradicted / unrelated / unclear 중 하나와 한 줄 이유를 사람이 기록합니다. 선택적으로 LLM judge를 초벌 검토에 쓸 수 있지만 model·rubric·원출력을 남기고 최종 표본은 사람이 확인합니다. Source retrieval은 답변에 실린 citation ID로 대신 계산하지 않습니다. Retriever의 원래 top-k 로그를 evaluator에 전달해야 “검색은 성공했지만 생성기가 근거를 선택하지 않은 실패”를 분리할 수 있습니다.

자동 확인

  • 기대 source가 top-k 안에 있는지
  • answered에 evidence가 하나 이상 있는지
  • quote가 source 원문에 정확히 존재하는지
  • 답 없음은 not_answered, 위험 요청은 needs_review인지
  • step/call/cost 상한을 넘지 않았는지

사람 확인

  • 답변이 질문에 직접 답하는지
  • quote가 존재할 뿐 아니라 주장에 실제로 관련되는지
  • quote가 답변의 핵심 조건·부정·예외와 모순되지 않는지
  • 필요한 조건이나 예외를 누락하지 않았는지
  • 거부 문구가 과도하게 모호하지 않은지

한 숫자로 모두 합치지 말고 source retrieval, exact quote integrity, semantic claim support, answer relevance, policy를 나누어 보고합니다. quote in source가 참이라는 이유만으로 “grounded” 또는 “사실”이라고 결론내리지 않습니다.

위험 측정·관리 원칙 참고: NIST AI RMF 1.0 및 Generative AI Profile (접속일: 2026-07-14). 프로젝트 rubric과 예제는 독자적으로 작성했습니다.

9. 실패 분석

실패를 다음 하나 이상으로 분류합니다.

  • corpus: 답이 없거나 문서가 오래됨
  • ingestion/chunk: 필요한 문장이 유실되거나 분리됨
  • retrieval: 기대 source가 top-k에 없음
  • generation: 근거는 찾았지만 답변이 왜곡됨
  • citation integrity: source 또는 quote가 검색 원문과 불일치
  • semantic support: quote는 정확하지만 답변 주장과 무관·모순되거나 조건을 충분히 지지하지 못함
  • policy: 답하면 안 되는 요청에 답하거나 정상 요청을 과도하게 거부
  • operation: timeout, API 오류, 비용/step 상한 초과

설정을 여러 개 동시에 바꾸지 않습니다. Baseline을 저장하고 한 가지 변경 후 같은 평가 세트를 다시 실행합니다.

10. 보안과 운영 범위

  • Key는 환경변수로 주입하고 notebook, JSONL, screenshot에 남기지 않습니다.
  • 외부 입력과 retrieved document를 신뢰하지 않는 데이터로 취급합니다.
  • 프로젝트 필수 범위는 읽기 전용입니다. 메일 발송, 파일 삭제, 외부 변경 tool은 넣지 않습니다.
  • 요청 수·token·step 상한을 실행 전에 정합니다.
  • 평가 로그에는 질문, source ID, 상태, 오류 범주를 남기되 secret과 개인정보는 제거합니다.

12. 단계별 체크포인트

체크포인트 A — 설계

  • 네 문장 범위, data card, 10개 평가 질문을 검토받습니다.
  • 이 단계 전에는 model API를 연결하지 않습니다.

체크포인트 B — Retrieval baseline

  • 답변 생성 없이 기대 source hit를 측정합니다.
  • 실패를 corpus/chunk/retrieval로 분류합니다.

체크포인트 C — Answer와 정책

  • evidence validation, 답 없음, 위험 요청 검토 전환을 통과합니다.
  • answered 사례의 주요 주장별 semantic support 검토표를 작성합니다.
  • 무근거 answered는 0건이어야 합니다.

체크포인트 D — 최종 비교

  • Baseline과 한 가지 개선 후 결과를 같은 질문으로 비교합니다.
  • 성공 사례뿐 아니라 최소 세 개 실패와 다음 개선 순서를 발표합니다.

13. 평가 rubric

  • 문제와 범위 15%: 한 업무로 제한되고 제외 범위가 명시됨
  • 데이터와 metadata 15%: 출처·이용 근거·버전·추적 가능성
  • retrieval과 evidence 25%: 기대 source, exact quote integrity, semantic claim support, 답 없음
  • workflow/agent 제어 20%: schema, 권한, 종료, 로그, 검토 전환
  • 평가와 실패 분석 20%: 10개 이상 질문, baseline 비교, 원인 분류
  • 실행 문서와 보안 5%: 재현 절차, key 비노출, 비용 상한

필수 기능이 빠지거나 조작된 evidence가 있으면 총점과 관계없이 수정 후 재검토합니다.
