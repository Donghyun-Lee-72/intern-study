Section 10. RAG를 평가하고 개선하기

예상 시간: 필수 120~150분 / 선택 심화 30~60분
선수 과정: Section 8~9
API 비용: 필수 실습은 API 호출 없음. LLM judge는 선택이며 필수 통과 조건이 아닙니다.

이번 Section의 위치

Section 9에서 인용 검사를 통과하는 RAG pipeline을 만들었습니다. 이제 한두 질문의 성공 화면이 아니라 고정된 질문 세트와 평가 규칙으로 변경 전후를 비교합니다. retrieval과 generation을 따로 평가해야 어느 설정을 고쳐야 하는지 알 수 있습니다.

핵심 질문

데모가 한 번 성공한 것과 시스템이 반복해서 안정적으로 동작하는 것을 어떻게 구분할까요?

학습 목표

  1. 평가 case에 question, expected source, reference claims, answerability를 기록합니다.
  2. source retrieval, exact quote integrity, semantic claim support를 서로 다른 검사로 구분합니다.
  3. baseline에서 설정 하나만 바꾼 candidate를 같은 데이터로 비교합니다.
  4. 답이 없는 질문의 무근거 답변과 답할 수 있는 질문의 과도한 거부를 함께 측정합니다.
  5. 자동 검사와 사람 rubric의 역할을 구분합니다.
  6. LLM-as-a-judge의 변동성과 편향을 설명합니다.

이번 Section의 필수선과 발전선

  • 필수선: 같은 6개 사례에서 baseline과 한 가지 변경만 비교하고, 질문별 실패 세 건을 설명합니다.
  • 발전선: LLM judge, 더 큰 평가 세트, 통계적 신뢰구간은 선택입니다.
  • 복습 신호: 평균 하나로 결론을 내리고 싶어진다면 질문별 결과로 돌아가 retrieval·상태·근거 중 어디가 변했는지 확인합니다.

용어를 많이 암기하는 장이 아닙니다. 여러 평가 개념을 완벽히 정의하는 것보다 “검색 실패와 답변 실패를 같은 점수로 섞지 않는다”와 “동시에 여러 설정을 바꾸지 않는다”를 실제 비교 결과로 보여주는 것이 필수입니다.

1. 시작 전 확인

  • Section 8의 Recall@k는 무엇을 측정했나요?
  • 정답 chunk가 검색되었는데 최종 답변이 틀릴 수 있는 이유는 무엇인가요?
  • 답변이 reference와 단어가 다르지만 의미는 같을 때 exact match는 어떻게 될까요?
  • 설정 두 개를 동시에 바꾸면 개선 원인을 알 수 있을까요?

2. 평가 단위를 먼저 고정하기

각 case는 최소 다음 필드를 가집니다.

{
  "case_id": "answerable-secret-leak",
  "question": "인증정보 유출이 의심될 때 첫 대응은 무엇인가요?",
  "answerable": true,
  "expected_source_ids": ["security-guide"],
  "required_claims": ["key를 폐기한다", "새 key를 발급한다"]
}

reference answer를 문장 하나로만 두면 같은 뜻의 다른 표현을 오답으로 처리하기 쉽습니다. 이 실습은 반드시 포함해야 할 핵심 claim을 함께 기록합니다. claim 역시 완전한 정답 판정기가 아니므로 자동 coverage와 사람 검토를 같이 사용합니다.

평가 데이터는 답할 수 있는 질문만 모으지 않습니다. 범위 밖 질문, corpus에 없는 일정 질문처럼 answerable=false인 case를 반드시 포함해야 무근거 답변률을 볼 수 있습니다.

3. Retrieval과 generation 평가 분리

질문별 로그는 다음 두 층으로 나눕니다.

Retrieval

  • 기대 source가 top-k 안에 있었는가?
  • 답이 없는 질문에서 threshold 이후 결과가 비었는가?
  • 어떤 chunk와 score가 반환되었는가?

Generation과 정책

  • 답변 가능 여부 상태가 맞는가?
  • 필요한 claim이 포함되었는가?
  • 모든 주요 claim이 retrieved context에 근거하는가?
  • 질문에 직접 답하며 무관한 내용을 늘리지 않았는가?

retrieval이 실패한 case에서 prompt만 바꾸는 것은 우선순위가 아닙니다. 반대로 정답 chunk가 top-k에 있는데 claim을 누락했다면 retriever 변경보다 generation prompt, context 구성 또는 output policy를 확인합니다.

특히 혼동하기 쉬운 세 단계

검사 묻는 질문 이 실습의 판정 방식
Source retrieval 기대한 source/chunk가 top-k에 들어왔는가? source ID로 자동 확인
Exact quote integrity 인용한 문장이 실제로 검색된 원문에 글자 그대로 있는가? chunk 범위와 substring으로 자동 확인
Semantic claim support 그 인용이 답변의 주요 주장을 실제로 지지하는가? 답변·주장·인용을 나란히 읽는 사람 검토

세 검사는 누적 관계지만 서로를 대신하지 않습니다. 기대 source를 찾았어도 엉뚱한 문장을 인용할 수 있고, 원문에 정확히 존재하는 문장을 인용했어도 답변의 결론과 무관하거나 오히려 모순될 수 있습니다. 따라서 이 Section에서 substring 검사는 인용 무결성만 증명합니다. 의미적 사실성이나 주장의 참을 증명한다고 표현하지 않습니다.

평가 개념과 retrieval/generation 분리 확인: LangSmith 공식 문서, Evaluate a RAG application (접속일: 2026-07-14). 평가 데이터와 코드는 독자적으로 작성했습니다.

4. 지표를 해석하는 최소 기준

이 실습에서 사용하는 값은 0 또는 1을 case별로 기록한 뒤 평균냅니다.

  • retrieval_success: 기대 source가 검색되었는가? 답 없음 case에서는 결과가 비었는가?
  • correct_status: answerable에 맞게 answered/not_answered를 선택했는가?
  • exact_quote_integrity: answered 응답에서 citation의 chunk가 검색되었고 quote가 해당 원문에 정확히 존재하는가?
  • required_claim_coverage: reference의 핵심 claim 중 답에 포함된 비율

semantic_claim_support는 주요 주장별 0/1/2 사람 rubric으로 따로 기록합니다. required_claim_coverage 역시 문자열 포함 여부를 보는 학습용 신호일 뿐, 동의어·부정·조건·모순을 올바르게 이해하는 의미 판정기가 아닙니다. 인용이 없는 not_answered/needs_review는 quote 검사 분모에서 제외하고 해당 없음으로 둡니다. 이를 성공 1점으로 세면 답변을 모두 거부하는 시스템이 인용 점수만 높아지는 왜곡이 생깁니다.

하나의 평균만 보고 결론내리지 않습니다. 예를 들어 threshold를 높이면 답 없는 질문의 retrieval specificity는 좋아질 수 있지만, 표현이 다른 정답 질문을 놓칠 수 있습니다. 전체 평균과 case별 실패를 함께 봅니다.

5. 수동 rubric

각 답변을 0/1/2점으로 사람이 평가하고, 점수 옆에 근거를 한 줄 적습니다.

  • Correctness: reference의 핵심 내용과 모순 없이 질문에 답하는가?
  • Semantic claim support: 답의 모든 주요 주장이 retrieved chunk로 지지되는가? 인용이 무관하거나 모순되지는 않는가?
  • Answer relevance: 질문에 직접 답하며 무관한 설명을 늘리지 않는가?

0은 핵심 실패, 1은 일부 충족 또는 애매함, 2는 충족입니다. 평가자가 다르면 일부 case를 겹쳐 채점하고 의견이 다른 이유를 기록합니다. rubric 문구를 바꾸었다면 이전 점수와 직접 비교하지 않습니다.

6. 필수 실습 A: baseline 저장

notebooks/10_rag_evaluation.ipynb는 Section 9의 고정 corpus와 6개 평가 case를 사용합니다. 먼저 baseline의 다음 값을 함께 저장합니다.

코드/데이터 버전
retriever 설정(k, threshold)
generator와 policy 버전
case별 retrieved chunk ID
raw answer와 status
  retrieval metric, exact quote validation error, semantic support 검토 기록

기본 실행은 파일을 쓰지 않습니다. 제출할 때만 SAVE_RESULTS=True로 바꿔 JSON을 만들고, 생성된 결과를 diff로 확인합니다. API 모델을 사용했다면 모델명, snapshot, 실행 시각, 재시도 횟수도 추가합니다.

7. 필수 실습 B: 한 가지 설정만 변경

제공 실습은 k=2를 유지하고 minimum_score만 바꿉니다. 일부러 지나치게 엄격한 candidate를 사용해, 답 없음 검색은 줄지만 답할 수 있는 질문의 근거까지 사라지는 부작용을 관찰합니다. 실행 전에 가설을 먼저 씁니다.

가설: threshold를 높이면 답이 없는 질문의 무관한 검색 결과가 줄어든다.
고정: corpus, chunk, 질문, k, generator, policy
변경: minimum_score 한 개
관찰할 부작용: answerable 질문의 정답 chunk 누락

결과가 나빠져도 실습 실패가 아닙니다. 가설과 다른 결과를 숨기지 않고 어떤 case가 왜 달라졌는지 설명하는 것이 목표입니다.

8. 최소 3개 실패 분석

실패를 다음 순서로 분류합니다.

  1. Corpus: 정답 내용이 원문에 없음
  2. Chunk: 정답 문장이 분할 과정에서 손상됨
  3. Query/retrieval: 기대 chunk가 top-k에 없음
  4. Context: 검색 결과가 모델 입력에서 누락되거나 잘림
  5. Generation: 근거를 받았지만 claim 누락·왜곡·잘못된 citation
  6. Policy: 답 없음/검토/답변 상태를 잘못 선택
  7. Evaluation: reference 또는 metric 자체가 부정확함

각 실패에는 관찰 증거, 가장 가능성 높은 원인, 다음에 하나만 바꿔볼 개선안을 적습니다. “모델이 나쁨”처럼 단계가 없는 설명은 원인 분석으로 인정하지 않습니다.

9. LLM-as-a-judge는 보조 도구

자유로운 자연어 답변의 correctness와 relevance를 사람이 전부 평가하기 어려울 때 judge 모델을 사용할 수 있습니다. 그러나 judge는 정답 판정기가 아닙니다.

  • prompt의 표현과 평가 순서에 영향을 받을 수 있습니다.
  • 같은 모델도 시점·설정·샘플링에 따라 결과가 달라질 수 있습니다.
  • 길거나 자신감 있는 답을 더 좋게 평가하는 편향이 생길 수 있습니다.
  • 생성 모델과 같은 계열 judge가 같은 오류를 공유할 수 있습니다.
  • private context를 외부 judge에 보내기 전에 데이터 정책을 확인해야 합니다.

따라서 judge를 쓰면 model/version, rubric prompt, raw rationale 또는 근거, 실행 시각을 남기고 작은 표본은 사람이 재검토합니다. judge 점수만 높았다는 이유로 공개하지 않습니다.

평가 설계의 일반 원칙 확인: OpenAI 공식 문서, Evaluation best practices (접속일: 2026-07-14). 본 Section의 metric과 rubric은 수업 목적에 맞게 독자적으로 작성했습니다.

10. 흔한 잘못된 비교

  • 질문 세트를 바꾼 뒤 baseline과 candidate 평균을 비교함
  • chunk size와 embedding model과 prompt를 동시에 바꿈
  • 성공한 질문만 골라 평균을 냄
  • 오류 난 API 호출을 조용히 제외함
  • LLM judge 점수를 ground truth처럼 사용함
  • 평균은 좋아졌지만 중요한 안전 case의 퇴행을 확인하지 않음

비교 전후에 동일한 case ID가 모두 존재하는지 assertion으로 검사하고, 실패·timeout·거부도 결과로 기록합니다.

11. 결과를 읽는 순서

  1. 실행되지 않았거나 누락된 case가 없는지 확인합니다.
  2. 답 없음 질문의 무근거 answered가 있는지 먼저 확인합니다.
  3. answerable 질문의 retrieval hit를 확인합니다.
  4. hit인데 generation이 실패한 case를 분리합니다.
  5. 평균 지표와 case별 퇴행을 함께 봅니다.
  6. 변경한 한 설정으로 설명 가능한 차이인지 판단합니다.
  7. 다음 실험에서도 한 가지 가설만 선택합니다.

12. 통과 기준과 제출물

  • 최소 6개 case에 answerable과 unanswerable이 모두 포함됩니다.
  • baseline과 candidate가 같은 case·corpus·generator를 사용합니다.
  • 변경한 설정이 하나뿐임을 코드와 기록으로 확인합니다.
  • source retrieval과 exact quote integrity를 별도로 출력합니다.
  • answered 사례는 주요 주장별 semantic support를 사람이 검토하고 이유를 기록합니다.
  • 최소 3개 실패를 단계별 원인과 개선 후보로 분류합니다.
  • 자동 점수 하나와 수동 rubric 하나의 불일치 사례를 설명합니다.

제출물은 comparison.json, 의미적 지지 0/1/2 수동 검토표, 가장 중요한 실패 3개의 분석, 다음 실험 가설 한 개입니다. secret이나 개인 경로가 결과 파일에 없는지 제출 전에 확인합니다.

13. 선택 심화

  • 두 평가자의 rubric 일치/불일치 비교
  • bootstrap confidence interval 또는 작은 표본의 불확실성 토론
  • query rewrite 전후 retrieval 성능 비교
  • 비용·latency를 품질 지표와 함께 기록
  • LLM judge와 사람 점수가 다른 case만 재검토
