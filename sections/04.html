Section 4: 여러 단계를 하나로 연결하기

예상 60-90분 Workflow Tool Agent-like flow

핵심 개념

Workflow는 여러 단계를 순서대로 연결한 실행 흐름입니다. Section 4에서는 검색, 답변 생성, 사실 추출, schema validation, 로그 기록을 하나로 묶습니다. 여기서 만드는 것은 완전 자율 agent가 아니라 agent-like workflow입니다. AI를 사용하지만, 어떤 단계를 어떤 순서로 실행할지는 사람이 명시적으로 정합니다.

초심자에게는 이 방식이 더 안전합니다. 완전 자율 agent loop는 모델이 다음 행동을 직접 고르고, 실행 결과를 관찰한 뒤 다시 다음 행동을 고르는 구조입니다. 강력하지만 디버깅이 어렵습니다. 반면 고정 workflow는 각 단계의 입력과 출력이 보이므로, retrieval이 틀렸는지, 답변 생성이 틀렸는지, schema validation이 틀렸는지 나눠서 확인할 수 있습니다.

Tool은 workflow가 호출할 수 있는 외부 기능입니다. 이번 실습에서는 검색 함수, tokenization 함수, percentage extraction 같은 작은 Python 함수가 tool에 가깝습니다. 좋은 tool은 입력과 출력이 좁고 명확합니다. 너무 넓은 analyze_everything 같은 함수보다, 작고 검증 가능한 함수가 workflow를 안정적으로 만듭니다.

이번 Section의 최종 그림

질문이 들어오면 먼저 관련 문서를 찾습니다. 찾은 문서에서 답변을 만들고, 답변에서 핵심 사실을 구조화합니다. 그 구조가 schema에 맞는지 확인하고, 마지막으로 어떤 단계가 실행되었는지 log를 남깁니다. 최종 출력에는 answer_ko, facts, evidence, not_answered, log가 함께 있어야 합니다.

이 구조는 나중에 더 자율적인 agent로 확장할 수 있습니다. 다만 지금 단계에서는 “AI가 알아서 다 한다”보다 “어떤 단계는 사람이 고정하고, 어떤 단계만 AI에게 맡긴다”를 구분하는 것이 더 중요합니다.

Section 4 결과 확인
  • dry run에서 질문별 retrieved document와 route가 출력되어야 합니다.
  • 최종 출력에 route, answer_ko, evidence, next_action_ko가 있어야 합니다.
  • 답이 없는 질문은 not_answered, 애매한 질문은 needs_review로 분기할 수 있어야 합니다.
  • workflow가 검색, 판단, 답변 단계를 분리한다는 점을 설명할 수 있어야 합니다.
  • workflow와 agent의 차이를 “누가 다음 행동을 고르는가”로 설명할 수 있으면 통과입니다.
